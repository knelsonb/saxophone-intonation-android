## 1. Goals (user spec, 1-paragraph restate)

v1.2 wires the metronome through the v1.1 TSF synth so clicks become real GM
drum samples. The user gains four new capabilities: (a) a **CUSTOM time
signature** — pick any numerator (1..32) and any denominator (2, 4, 8, 16,
32) so 5/8, 7/16, 13/4 are all expressible; (b) a **per-beat pattern editor**
where each beat in the bar has its own drum voice, default kick-on-1
click-on-rest; (c) **subdivisions** (8ths, 16ths, triplets) that can be
toggled on with their own voice; (d) a **full GM drum picker** modeled on the
v1.1 voice-picker (5 quick-picks + expandable full ~47-entry list). All of
this lives in SETUP under a single METRO group, with METRO-tab additions
limited to what a player actually touches mid-session.

## 2. Where each control lives (Tab Placement Decisions)

This is the load-bearing section. Get this wrong and the METRO tab needs a
ScrollView, which violates a hard layout rule.

### METRO tab (no ScrollView, must fit on Pixel-9-Pro-class screen)

- BPM display + ±1 / ±5 flank steppers (unchanged from v1.1).
- Beat visualization (PULSE / FLASH / PENDULUM — unchanged).
- Time signature row: now **5 pills** — `2/4 3/4 4/4 6/8 CUSTOM`. The
  `CUSTOM` pill replaces nothing; the row already has `flexWrap:'wrap'` so a
  5th pill flows safely on narrow phones.
- Inline custom-signature stepper: shown **only when `CUSTOM` is the active
  pill**. One row, two compact ± steppers (numerator / denominator) plus a
  read-only "5/8" preview in the middle. Sits directly under the time-sig
  pill row. Reserves zero height when CUSTOM is not selected (see section 10
  on flicker discipline).
- TAP button (unchanged).
- START / STOP (unchanged).

That is it for the METRO tab. The per-beat pattern editor, subdivision
checkboxes, drum kit picker, and full-GM drum browser **DO NOT** live on
METRO. A saxophone player who has the metronome running does not want to tap
into a 47-row picker to pick a snare voice — they want to glance at the
tempo and breathe.

### SETUP tab, new METRO group (`METRO VOICES`)

A single new `settingsGroup` titled `METRO VOICES`, placed **directly under
the existing METRO CALIBRATION group**. The order: METRO STYLE, METRO
CALIBRATION, **METRO VOICES (new)**, DECK STYLE, DRONE VOICE. Sequencing
chosen because METRO STYLE → METRO CALIBRATION → METRO VOICES keeps all
metronome-related groups contiguous.

The new group contains, in order:

1. **Default pattern toggle** — `DEFAULT KIT (kick + click)` vs `CUSTOM
   PATTERN`. Two-pill toggle, same idiom as `gainPill` (used by output
   route, filter mode, etc).
2. **Per-beat row** (visible only when CUSTOM PATTERN is selected) — N cells
   (N = current numerator) showing beat number + current drum label. Tap a
   cell to open the drum picker sheet. See section 4 for layout rules.
3. **Subdivisions** — a four-way pill row: `OFF · 8TH · 16TH · TRIPLET`.
   Mutually exclusive (radio).
4. **Subdivision voice** (visible only when subdivisions are not OFF) — a
   single drum-picker row labeled `Sub voice` showing the current sub
   instrument; tap to open the same drum picker sheet.
5. **Full GM drum browser** (collapsed by default) — `More drums · 47 GM
   percussion ▼`. Pure browse: tapping a row sets the **default sub voice**
   if the picker was opened from the sub voice control, or the
   **most-recently-edited beat cell** if opened from a beat cell. (The
   picker sheet remembers which control invoked it.)

### Drum picker — modal sheet (`@gorhom/bottom-sheet` `BottomSheetModal`)

Following the established BottomSheetModal idiom (PitchPipes.tsx,
IntonationTable.tsx). 92% snap point. Layout inside the sheet:

- 5-up quick-pick pills (kick / snare / hh-closed / hh-open / click).
- Inline header showing the **currently selected** drum + GM note number.
- Scrollable list of all ~47 GM percussion notes (MIDI 35..81).
- Tapping a row dismisses the sheet and writes the choice back to whichever
  cell or control opened it.

Why a sheet rather than inline expansion: the per-beat row already lives
inside SETUP's ScrollView. An inline 47-row expander nested in a parent
ScrollView gives the v1.1 setupVoiceListWrap nested-scroll issue — workable
but cramped. A sheet at 92% gives the user a real surface to browse on, with
backdrop dismiss. The drum picker is also re-opened from multiple sites
(each beat cell, each sub voice slot) so centralizing it in one sheet
component is cleaner than five inline expanders.

### Justification recap (against the hard rules)

- **METRO tab fits one screen**: only the CUSTOM signature stepper is new on
  this tab and it costs at most ~52dp when visible, reclaimable from the
  flex:1 visualization wrapper. When CUSTOM is not selected, height cost is
  zero.
- **No new ScrollView on METRO**: confirmed. All scrolling-required UI
  (per-beat row, GM drum list) lives in SETUP or the bottom sheet.
- **Tab bar stays put**: no changes touch the bottom tab strip.
- **SETUP is scrollable + persistent scrollbar**: confirmed in current
  SetupScreen. The new group inherits this automatically.

## 3. Custom time-signature flow

### Wireframe (METRO tab, CUSTOM active)

```
   [−5] [−1]   120   [+1] [+5]
   ┌─────────────────────────────────┐
   │  (PULSE / FLASH / PENDULUM)     │
   │              · ·                │
   │            · · · ·              │
   └─────────────────────────────────┘
   [2/4] [3/4] [4/4] [6/8] [CUSTOM]      ← CUSTOM is active
   ┌─────────────────────────────────┐
   │  NUM [−][ 7 ][+]   DEN [−][8][+]│   ← only shown when CUSTOM active
   │            7/8                  │   ← live preview
   └─────────────────────────────────┘
              [   TAP   ]
              [  START  ]
```

### Interaction

- Tap `CUSTOM`. The custom-signature row appears beneath the pill row.
  Initial values: numerator = last-used custom num (default 5), denominator
  = last-used custom den (default 8). Persisted separately from preset
  selection so that toggling between preset and CUSTOM never loses the
  user's custom value.
- Stepping the numerator: ±1 per tap. Clamps to [1, 32]. Wrap is OFF
  (stepping past the edge is a no-op; the tap still gives haptic-style
  press feedback).
- Stepping the denominator: cycles through **`[2, 4, 8, 16, 32]`** only.
  These are the standard musical denominators. No `3` or `5` — that is what
  the numerator is for; a denominator must be a power of two musically.
  Steps are `−` = prev in list, `+` = next in list. Edge taps are no-op
  (no wrap).
- Tap any preset pill again to leave CUSTOM. The custom-signature row
  disappears; the running click immediately switches to the preset's
  beat-per-bar count. Custom num/den are remembered.
- The live preview text in the middle of the stepper row shows e.g. `7/8`
  with `fontVariant: ['tabular-nums']` so the layout doesn't shift when
  numerator goes from 9 to 10.

### TimeSig type extension

Today `TimeSig` is `'2/4' | '3/4' | '4/4' | '6/8'`. Replace with a tagged
union:

```ts
export type TimeSig =
  | { kind: 'preset'; value: '2/4' | '3/4' | '4/4' | '6/8' }
  | { kind: 'custom'; num: number; den: 2 | 4 | 8 | 16 | 32 };
```

Helpers:

```ts
export function timeSigBeats(s: TimeSig): number {
  if (s.kind === 'preset') return TIME_SIG_BEATS[s.value];
  return s.num; // beats-per-bar = numerator regardless of denominator
}

export function timeSigLabel(s: TimeSig): string {
  return s.kind === 'preset' ? s.value : `${s.num}/${s.den}`;
}
```

The denominator currently does NOT affect click cadence — useMetronome
schedules on `60000 / bpm` per beat regardless of denominator. **Keep that
contract**. The denominator is for notation and (future) feel-aware
subdivision counting. Document this loudly in the type's JSDoc so a future
engineer doesn't try to make 6/8 sound like 6/8.

(The user's 6/8 preset today already counts six eighth-notes per bar at the
displayed BPM. That is the established behavior. Custom inherits it.)

### Bounds + edge cases

- `num < 1`: stepper clamped; can never reach 0.
- `num > 32`: stepper clamped. 32 is sane — anything beyond is a notational
  curiosity (32/4 is a kind of writing exercise) and would also make the
  per-beat row layout (section 4) impractical even with horizontal scroll.
- `den` is the closed set `{2, 4, 8, 16, 32}`. Never anything else.
- Switching from `4/4` to `7/8` while the metronome is running: scheduler
  resets bar position (next fired beat becomes the new downbeat "1"), same
  as the existing `setTimeSig` does. Pattern array (section 4) is resized:
  if shrinking, drop overflow cells; if growing, fill new cells with the
  default click voice. Existing cells keep their user-assigned voice.
- Resizing the pattern is a non-destructive merge: dropping cells does NOT
  lose the kick-on-1 if the user-assigned beat 1 voice gets dropped (it
  wouldn't be — beat 1 is always present unless num=0, which we don't
  allow).

## 4. Per-beat pattern + drum picker

### Default pattern

Per the user spec: kick on beat 1, click on beats 2..N.

```ts
// GM drum note numbers (channel 9):
const GM_KICK_ACOUSTIC   = 36; // "Bass Drum 1"
const GM_CLICK_METRONOME = 76; // "High Wood Block"  — see section 11 Q1
```

Default `pattern[]` for any new numerator N:

- `pattern[0]   = { midi: 36, velocity: 110 }` (kick, accent)
- `pattern[i>0] = { midi: 76, velocity:  90 }` (click)

This is the **fresh-install** default. The user's existing accent/normal
click WAVs from v1.1 are NOT used in v1.2 once the user is on the new path.
See section 7 for migration: existing v1.1 users get the same default
pattern computed for their current numerator the first time they open v1.2.

### Layout: per-beat row in SETUP

Live in the `METRO VOICES` group, under the `DEFAULT KIT / CUSTOM PATTERN`
toggle, visible only when `CUSTOM PATTERN` is selected.

Each cell is a `Pressable`:

- Width: `flex: 1` within the row up to a max of 60dp; min 44dp (touch
  target).
- Height: 56dp (matches `metroTap` height, gives finger room).
- Content (centered, 2 lines):
  - Line 1: beat number, e.g. `1`, `2`, ..., `N`. `tabular-nums`,
    `fontSize: 12`, `letterSpacing: 1`, `color: inkDim`.
  - Line 2: drum short label, e.g. `KICK`, `SNARE`, `HH-C`, `CLICK`,
    `RIDE`. `fontSize: 11`, `letterSpacing: 1`, `fontWeight: '700'`.
- Selected/edited cell (most recently edited): subtle accent border
  (`borderColor: accent`) so the user knows which cell opens when they go
  back to the sheet.
- Beat-1 cell gets a permanent `borderLeftWidth: 3, borderLeftColor:
  accent` so the user can always see where the downbeat is in any
  numerator. (No marching-band confusion.)

### Layout: N varies from 1 to 32

This is where the design earns its keep.

- **N ≤ 8**: cells fit in a single row across the SETUP width. Plain
  `flexDirection: 'row'` with `gap: 6`.
- **9 ≤ N ≤ 16**: cells fit in **two rows of 8 each** via `flexWrap:
  'wrap'`. We deliberately reduce the per-cell content (drop the drum
  label, show just the beat number; tap reveals voice in the sheet) so the
  row fits without horizontal scroll on a 360dp-wide phone.

  Wait — that loses the voice-at-a-glance affordance. Better: keep the
  drum label, but compress label fontSize from 11 to 10, and let cells
  hit min 44dp. 8 × 44 + 7 × 6 = 394dp. Phone widths run ~360–412dp. Too
  tight on the narrow end.

  **Final rule for N > 8**: row uses horizontal `ScrollView` (showsHorizontalScrollIndicator,
  `nestedScrollEnabled`), and we keep a fixed 56dp-wide cell width. User
  swipes the row to see beats 9..N. This is the LEAST surprising idiom —
  the row visually communicates "there's more, scroll" via the indicator,
  and beat 1 is always visible at the start.
- **N > 16**: same horizontal scroll, just longer. No special case at 16.
  We never wrap to a second row — wrapping breaks the visual cadence
  ("these all belong to the same bar"). One bar = one row.

### Tapping a cell

Opens the **drum-picker bottom sheet** (section 6 details the sheet UI),
pre-scrolled to the currently selected drum. The sheet remembers which cell
opened it via local state in the picker host component (a `useState<
{ kind: 'beat'; index: number } | { kind: 'sub' } | null >`). On selection
the sheet writes the new midi note + velocity back to `pattern[index]` and
dismisses.

### Per-beat velocity

Out of scope for v1.2 UI surface — but the type carries it (`{ midi,
velocity }`) so a future v1.3 can expose per-cell velocity without a state
migration. Default velocities: 110 for beat 1, 90 for beats 2..N. The drum
picker sheet does not let the user edit velocity in v1.2. Capture this in
section 11 as a deferred decision.

### Long-press a cell

**Don't add a long-press in v1.2.** It is non-discoverable and the user's
spec does not ask for it. If a future "reset this cell to default" need
arises, expose it from the picker sheet (a `Reset cell` button).

## 5. Subdivisions checkboxes

Per the user spec: 8th notes, 16th notes, OR triplets within each beat.

### UI — radio, not multi-select

- **Mutually exclusive**, four states: `OFF · 8TH · 16TH · TRIPLET`. A
  fifth "8TH+16TH" state is musically incoherent for a metronome
  (overlapping ticks at different rates) and not in the spec.
- Idiom: same `gainPill` row used for output route and metro style. Four
  pills, one active. Already-discoverable visual style.
- Lives in SETUP under `METRO VOICES`, between the per-beat row and the
  full GM browser.

### Sub-tick scheduling rules

- `OFF`: scheduler fires once per beat (today's behavior).
- `8TH`: scheduler fires twice per beat — 1, &, 2, &, 3, &, ... The "&" is
  the sub-tick.
- `16TH`: four times per beat — 1 e & a, 2 e & a, ...
- `TRIPLET`: three times per beat — 1 tri-pl-et, 2 tri-pl-et, ...

Wall-clock target for sub-tick k (0-indexed within a beat, k=0 is the
downbeat itself):

```
subDelta = (60000 / bpm) * (k / subdivisionsPerBeat)
targetMs = startedAtMs + beatIndex * intervalMs + subDelta
```

The downbeat (k=0) **always uses `pattern[beatInBar - 1]`** (its
user-assigned voice). Sub-ticks (k>0) use `subdivisionVoice` (section 5
below).

### Subdivision voice — same for all sub-ticks, ALWAYS independently
assignable

The user spec says "Each sub-tick can also have its own instrument (or
inherit the parent beat's)" — this is two competing design choices. **Pick
ONE in v1.2**: a single global subdivision voice, independently assignable.
Rationale:

- Per-sub voice (3 different drums for triplet ticks 2/3/4) is a real
  rhythm-programming feature, not a metronome feature. The user's spec
  framing ("kick on the downbeat and a click on the other downbeats")
  treats the metronome as accent-marking, and per-sub-voice introduces a
  lot of UI for limited day-to-day value.
- Inheriting the parent beat means a 7/8 with kick-snare-click-click-...
  pattern + 8th subs would alternate kick-kick-snare-snare-click-click-...
  which is rarely what anybody wants for a click track.
- Independent single voice gives the user `kick + snare on beats, closed
  hi-hat on every &` — exactly the standard rock-drum-machine pattern.

Implementation: `subdivisionVoice: { midi: number, velocity: number }`,
default `{ midi: 42, velocity: 70 }` (closed hi-hat, quieter than the
beats). Surfaced in SETUP as the `Sub voice` drum-picker row described in
section 2.

(Capture per-sub voice as deferred decision in section 11. If user asks for
it later, the existing pattern-array idiom extends naturally to a 2D shape.)

### Default subdivision voice

**Closed hi-hat (GM 42), velocity 70.** Industry-standard click-track
ghosting voice. Quieter than the beats so the downbeat reads cleanly.

### When subdivisions are OFF

Hide the `Sub voice` row (height reserves 0 — `display: 'none'` is fine
because adjacent groups absorb the change cleanly; this is in SETUP's
ScrollView, not METRO, so flicker rules are relaxed here).

## 7. State machine extensions to useMetronome

### New shape

```ts
export type TimeSig =
  | { kind: 'preset'; value: '2/4' | '3/4' | '4/4' | '6/8' }
  | { kind: 'custom'; num: number; den: 2 | 4 | 8 | 16 | 32 };

export interface BeatInstrument {
  midi: number;       // GM channel-9 note 35..81
  velocity: number;   // 1..127, drives TSF noteOn velocity
}

export type Subdivision = 'off' | '8th' | '16th' | 'triplet';

export interface MetronomeState {
  // --- v1.1 (unchanged) ---
  bpm: number;
  setBpm: (n: number) => void;
  bumpBpm: (delta: number) => void;
  running: boolean;
  start: () => void;
  stop: () => void;
  toggle: () => void;
  registerTap: () => number | null;
  beat: number;
  pulse: number;
  clickVolume: number;
  setClickVolume: (v: number) => void;

  // --- v1.2 ---
  timeSig: TimeSig;
  setTimeSig: (s: TimeSig) => void;
  /** Per-beat pattern, length === timeSigBeats(timeSig). */
  pattern: BeatInstrument[];
  setBeatInstrument: (index: number, voice: BeatInstrument) => void;
  /** 'default' = use the kick+click defaults; 'custom' = use pattern array. */
  patternMode: 'default' | 'custom';
  setPatternMode: (m: 'default' | 'custom') => void;
  /** Last-used custom numerator/denominator — persisted independently
   *  so toggling between preset and custom doesn't lose the user's value. */
  customNum: number;
  customDen: 2 | 4 | 8 | 16 | 32;
  setCustomNum: (n: number) => void;
  setCustomDen: (d: 2 | 4 | 8 | 16 | 32) => void;
  /** Subdivision setting. */
  subdivision: Subdivision;
  setSubdivision: (s: Subdivision) => void;
  /** Voice used for every sub-tick. */
  subdivisionVoice: BeatInstrument;
  setSubdivisionVoice: (v: BeatInstrument) => void;
  /** Reserved for v1.3 kit selection. v1.2 always 'default'. */
  kit: 'default' | string;
}
```

### Scheduler changes

The visual peak still fires at `targetMs` for the **downbeat** (sub-ticks
do NOT bump the `beat` counter — they fire audio only). This preserves
beat-1..N display semantics and the existing PULSE/FLASH/PENDULUM visuals
without modification.

For each beat that gets scheduled:

1. Resolve the voice for the downbeat:
   - If `patternMode === 'default'`: voice = (beatInBar === 1) ? KICK :
     CLICK with default velocities.
   - If `patternMode === 'custom'`: voice = `pattern[beatInBar - 1]`.
2. Fire the downbeat: send a TSF noteOn on channel 9 with that midi +
   velocity, pre-rolled by `routeLatency - clickOffsetMs` exactly as the
   current click WAV scheduling does. (Sauron owns the audio plumbing.)
3. If `subdivision !== 'off'`: schedule (k = 1..subsPerBeat-1) additional
   noteOn timers at `targetMs + k * intervalMs / subsPerBeat`, using
   `subdivisionVoice`.

### Audio implementation note (Sauron's territory, scoped here)

The v1.1 `expo-audio` click-WAV path is **replaced** by TSF noteOn on
channel 9. Drop the buildClickWavBase64 calls and the accent/normal
AudioPlayer refs once TSF can be triggered from the metronome scheduler. If
TSF isn't ready (still warming up, fallback path), the metronome should
**silently fall back to the v1.1 WAV clicks for one bar** rather than going
silent. Document this in the audio plumbing PR. Out of scope to design the
fallback here.

### clickVolume in v1.2

Reused. The v1.1 `clickVolume` (0..1) is multiplied into the TSF noteOn
velocity at the audio layer: `effectiveVelocity = round(velocity *
clickVolume)`, clamped to [0, 127]. A `clickVolume` of 0 mutes the click —
preserved.

## 8. Default values + migration

### Fresh install defaults

| Field              | Default                                          |
|--------------------|--------------------------------------------------|
| `timeSig`          | `{ kind: 'preset', value: '4/4' }`               |
| `patternMode`      | `'default'`                                      |
| `pattern`          | `[kick(36,110), click(76,90), click(76,90), click(76,90)]` (length 4) |
| `customNum`        | `5`                                              |
| `customDen`        | `8`                                              |
| `subdivision`      | `'off'`                                          |
| `subdivisionVoice` | `{ midi: 42, velocity: 70 }` (closed hi-hat)     |
| `kit`              | `'default'`                                      |

### Migration from v1.0 / v1.1 prefs

Old prefs schema had `timeSig: '2/4'|'3/4'|'4/4'|'6/8'` as a top-level
string. v1.2's `AppPrefs` adds:

```ts
// in AppPrefs
timeSigKind: 'preset' | 'custom';
timeSigPreset: '2/4' | '3/4' | '4/4' | '6/8';  // populated when kind=preset
customNum: number;                              // last-used, default 5
customDen: 2 | 4 | 8 | 16 | 32;                 // last-used, default 8
patternMode: 'default' | 'custom';
patternJson: string;                            // serialized BeatInstrument[]
subdivision: 'off' | '8th' | '16th' | 'triplet';
subVoiceMidi: number;
subVoiceVelocity: number;
```

`loadPrefs` migration:

- If `timeSigKind` missing but `timeSig` (legacy field) present: set
  `timeSigKind = 'preset'`, `timeSigPreset = legacy timeSig` (default
  `'4/4'`).
- If `patternJson` missing or empty: omit; the hook builds the default
  pattern from current numerator on mount.
- If `subdivision` missing: default `'off'`.
- Unknown values follow the `asOneOf` / `asInt` tolerant-coercion pattern
  already established in prefs.ts. Bad JSON in `patternJson`: silently
  reset to defaults rather than crash.

The legacy `timeSig` field stays in `AppPrefs` for one minor release for
backward read compat, marked `@deprecated`. `savePrefs` writes both old

## 10. Hard "do not screw this up" notes

### RESPECT these patterns (load-bearing)

- **Height reserves for inline-toggleable rows.** The METRO-tab custom
  signature stepper row, when CUSTOM is NOT selected, reserves zero height
  (`display: 'none'` is fine here; the surrounding `flex:1` visual
  wrapper absorbs the change because there is no rapid toggle). When
  visible, it has a fixed minHeight to prevent flicker as digits change
  width (use `tabular-nums`).
- **Beat-cell content stability.** Cells must use `tabular-nums` for the
  beat number and a **fixed minWidth for the drum short label** so the
  cell doesn't reflow when the user changes `CLICK` to `SIDE-STICK`.
  Truncate with `numberOfLines={1}` + `ellipsizeMode="tail"`.
- **Persistent scrollbar in SETUP** (the existing `persistentScrollbar`
  prop on the outer ScrollView). Keep it.
- **Nested-scroll cap.** When the GM drum browser expands inline (the
  "More drums" expander), wrap its inner scroll in a `maxHeight: 360`
  container exactly like `setupVoiceListWrap` does. **Don't let the inner
  list grow unbounded inside the outer SETUP ScrollView.**
- **Drum picker as a single bottom sheet**, not a per-cell modal. One
  component instance, one ref, parameterized by the `pickerTarget` state.
  Five inline expanders = five sources of bugs.
- **Beat-1 is visually distinct.** Always show a left accent stripe on the
  beat-1 cell, regardless of which drum is assigned. Saxophone players in
  odd time signatures need this.
- **`gainPill` for radio choices.** Reuse the existing pill row idiom for
  patternMode toggle, subdivisions, and the drum-picker quick-picks. Don't
  invent a new radio style.
- **TimeSig switch resets bar position, preserves phase.** Same as today's
  `setTimeSig`. Don't regress that.
- **AppState background pause** still works (the existing `wasRunningRef`
  pattern). New state must not break it.

### AVOID these anti-patterns

- **No ScrollView on the METRO tab.** Not horizontal, not vertical, not
  anywhere. The per-beat row's horizontal scroll is in SETUP, not METRO.
- **No long-press menus.** Non-discoverable. Tap = open picker. Done.
- **No nested bottom sheets.** If the picker sheet needs to confirm
  something, use an inline confirm row inside the sheet, not a second
  sheet on top.
- **Don't push the START/STOP button off-screen.** The custom signature
  stepper has a fixed height (estimate ~52dp); the existing PULSE/FLASH/
  PENDULUM wrappers are `flex: 1` so they absorb the cost. **Test on a
  720dp-tall (Pixel 4a class) phone before merge.**
- **Don't make `den` change beats-per-bar.** Beats-per-bar = numerator
  only, always. Denominator is for notation. (Capture in JSDoc; future
  engineers WILL try to fix this.)
- **Don't add per-sub voice in v1.2.** Single subdivisionVoice only.
- **Don't strip out the v1.1 click-WAV fallback path** until TSF metronome
  is verified solid on real hardware. Keep both paths in code; the
  fallback runs if `engine.tsfReady === false`. Drop in v1.3.
- **Don't auto-open the picker on first CUSTOM selection.** When the user
  taps CUSTOM PATTERN, just reveal the per-beat row. They tap a cell when
  they're ready. Auto-opening a sheet ambushes the user.
- **Don't reset the user's custom pattern when they toggle DEFAULT KIT
  back on.** Keep the pattern array in state; just stop reading from it.
  Toggling back to CUSTOM PATTERN restores their work.
- **Don't reset `customNum/customDen` when the user picks a preset pill.**
  Same rationale — the values are remembered separately.

## 11. Open questions for the user

Numbered so each is signable.

1. **The "click" drum default voice — GM 76 (High Wood Block) or GM 56
   (Cowbell)?** Both are standard click-track substitutes; 76 is closer to
   the v1.1 synthetic click sound, 56 is iconic but louder. **Frodo
   recommends 76** because it preserves the v1.1 feel — users who don't
   touch the new voice controls hear roughly the same metronome they had
   before. *Decision needed: 76 (Wood Block) or 56 (Cowbell)?*

2. **Subdivisions: confirmed mutually exclusive (radio)?** Or do you want
   to allow `8TH + TRIPLET` simultaneously (creates a polyrhythmic feel)?
   **Frodo recommends radio (one of OFF/8TH/16TH/TRIPLET).** Polyrhythms
   are a different feature and the spec phrasing reads as "OR." *Decision
   needed: radio or multi-select?*

3. **Per-subdivision voice: single global, or one voice per sub-tick
   index?** Section 5 picks single global to keep v1.2 scope sane. *Confirm:
   ship v1.2 with single sub voice; defer per-sub-index voice to v1.3?*

4. **Per-beat velocity in v1.2 UI?** Spec doesn't ask. Frodo recommends no
   UI surface in v1.2, but the type carries velocity so v1.3 can expose
   it. *Confirm: type carries velocity, UI hides it for v1.2?*

5. **`MAX_NUMERATOR` = 32?** Or should we cap at 16? Anything above 16
   means horizontal scroll in the beat-cell row. *Decision needed: 32 (per
   spec "1..32 or sane cap") or 16?*

6. **Behavior when `CUSTOM` is selected and numerator happens to equal a
   preset value (e.g. CUSTOM 4/4)?** Two valid options: (a) treat as
   custom regardless — user explicitly chose CUSTOM; (b) silently auto-
   demote to the matching preset. **Frodo recommends (a) — never silently
   change the user's selection.** *Confirm (a)?*

7. **TSF audio fallback during warm-up: degrade to v1.1 WAV clicks, or
   stay silent?** Frodo says degrade — the user pressing START expects
   sound, and a one-bar mismatch is far less surprising than no audio.
   *Confirm?*

8. **Drum picker GM list ordering: ascending MIDI number (35..81) or
   grouped by family (drums, cymbals, perc, latin, effects)?** v1.1 drone
   uses ascending program number. **Frodo recommends ascending MIDI** for
   consistency. *Decision needed: ascending or grouped?*

9. **Default sub-tick velocity: 70 (current proposal), or proportional
   (e.g. half the parent beat velocity)?** Proportional sounds nicer but
   adds coupling. *Confirm 70?*

10. **Should the v1.1 click-WAV `clickVolume` (0..1) preference also apply
    to TSF noteOn velocities in v1.2 (proposal: yes, multiplicative), or
    should v1.2 introduce a new `metroDrumVolume` and deprecate
    clickVolume?** Frodo recommends keep one knob (clickVolume) and
    multiply it through to velocity — fewer settings, same effect. *Confirm?*

11. **Do you want a UI affordance to "preview" a drum voice from the picker
    sheet (tap-and-hold to audition, or a small play-icon column)?** Spec
    doesn't ask. Without preview, the user picks → dismisses → next
    metronome tick reveals the sound. Frodo recommends skipping preview
    in v1.2 to keep scope tight. *Confirm: no preview?*

12. **Visual style on the per-beat row when the metronome is currently
    running**: should the cell of the currently-firing beat flash in sync
    with the audio? It's discoverable and beautiful but adds a tie to the
    `pulse` counter from inside SETUP. Frodo recommends **yes, but only
    if it adds <1 frame of overhead** — implementation should be a CSS-
    style accent border on the live cell, driven by `pulse` modulo
    numerator. *Decision needed: skip in v1.2, or include?*

## 14. Tab-aware TopBar

The current `TopBar` (`src/tunerWidgets.tsx:208`) renders the same chrome
on every tab: BELLCURVE wordmark → row2 (instrument badge, A= stepper,
PAGE/CONCERT pill, status dot) → optional status caption → TABLE/PIPES
row. Everything between the wordmark and the status row is TUNER-specific.
A user on METRO/DECK/SETUP eats ~110dp of tuner chrome they cannot act on
from that tab.

This section specifies a tab-aware redesign. Pass an `activeTab: TabKey`
prop in; switch the middle/bottom rows by tab; keep the always-on rows
identical across tabs so the header never reflows.

### 14.1 Always-on row (every tab, identical pixel positions)

Two elements survive on all four tabs and **must occupy the same y-offset
and the same height regardless of which tab is active**, so transitioning
between tabs doesn't shift the body content below.

- **BELLCURVE wordmark** (`styles.brand`). Identity. No tab info encoded.
- **Status dot + status caption**, lifted out of the TUNER row2 into its
  own dedicated row that renders on every tab. The status dot sits at the
  right edge of the wordmark row (so a "mic warming up" indicator never
  disappears when the user navigates to DECK or METRO).
- The status caption row (`WARMING UP`, `NO MIC`, `NO AUDIO`,
  `WAITING FOR MIC`) reserves its 16dp slot even when listening is
  healthy and the caption is null — this kills the cross-tab reflow that
  happens today when status changes mid-session.

Per section 14.4 below, mic status matters on every tab (DECK records
from the mic; TUNER + METRO calibration depend on it; SETUP shows the
audio-source toggle that the user reaches for *because* status went red).

### 14.2 Per-tab pill set

Reuse the existing `metroSigPill` / `displayPill` / `iconBtn` style keys
where possible. Don't invent new chrome — each tab picks from the
existing vocabulary.

#### TUNER (status quo, slightly trimmed)

- Instrument badge (`onBadgePress` → instrument picker)
- A= stepper (refHz)
- PAGE / CONCERT toggle
- TABLE / PIPES nav row

Justification: this is what's there now and it works. Don't fix what
isn't broken. The status dot moves up to the always-on row (14.1) but
the pill set is otherwise unchanged.

#### METRO (lean — body shows the big stuff)

- **Instrument badge** (read-only label; tap goes to picker — same as
  TUNER). Saxophone player still wants to confirm "yes I'm in tenor
  mode" from any tab.
- A= stepper. Reference pitch matters for the metronome's TSF voices in
  v1.2 (they pitch the click WAV-replacements at A=440 by default; if the
  user has A=442 they want the click sympathetic — Sauron's call whether
  to apply, but the affordance belongs here).
- **No PAGE/CONCERT toggle.** The metronome doesn't render notes. The
  toggle has zero effect on METRO and confuses the user.
- **No TABLE/PIPES nav row.** TABLE is intonation log, PIPES is reference
  pitch — both are TUNER navigation. If a user wants PIPES they switch to
  the TUNER tab. Keeping them on METRO clutters and implies the
  metronome interacts with them. It doesn't.
- **No BPM duplication.** METRO body shows BPM at fontSize 64 — putting
  it in the top bar at fontSize 14 is noise. Skip.
- **No tap-tempo in top bar.** METRO body has the TAP button. Skip.

Net: METRO TopBar = wordmark + status row + badge + A= stepper. Two
rows total below the wordmark vs. four on TUNER.

#### DECK (minimal — recording UI is on the body and the tab strip)

- **Instrument badge** (consistent across tabs — the user identity).
- A= stepper. The DECK records the user's playing; the tuning reference
  applied at record time should be visible.
- **No PAGE/CONCERT toggle.** Same reasoning as METRO.
- **No TABLE / PIPES.** Same.
- **No recording-elapsed timer.** TabBar already shows the red recording
  dot; DeckScreen body shows the elapsed time. Don't double-render.

Net: DECK TopBar = identical row shape to METRO. Badge + A= only.

#### SETUP (minimal)

- **No instrument badge** in the top bar. Instrument selection is a
  SETUP row itself; showing it in the header AND in the body
  (potentially with a different label) invites the user to wonder why
  there are two affordances.
- **No A= stepper** in the top bar. SETUP has TUNING REFERENCE as its
  own group; same double-affordance problem.
- **No PAGE/CONCERT, no TABLE/PIPES.**

Net: SETUP TopBar = wordmark + always-on status row. Pure chrome. Tabs
below tell the user where they are; the page itself is fully
self-describing.

### 14.3 Layout consistency — fixed header height

This is the load-bearing rule.

- The TopBar's total height is **identical across all four tabs**. We
  achieve this by giving the per-tab pill area a fixed `minHeight`
  matching the tallest tab (TUNER, which has the most rows). For tabs
  with fewer rows (DECK, SETUP), the area renders empty space — not a
  visible "gap," just unfilled height that the body content below sits
  consistently beneath.
- Equivalently: render an invisible spacer `View` per-tab that pads up
  to the TUNER row count. Same anti-flicker pattern as the v1.1
  NoteReadout fix.
- **Why this matters**: today, navigating TUNER → METRO leaves the
  TopBar shorter (no TABLE/PIPES row gets rendered conditionally on
  some screens; if we made the TopBar conditionally shorter on METRO/
  DECK/SETUP, the body jumps up ~88dp on tab change). Avoid that. Tab
  navigation must feel like switching content, not redrawing chrome.
- Pixel budget (Pixel 9 Pro class, dp): wordmark row 28 + status row
  16 + pills area 88 (reserved on every tab) + 4 padding = 136dp total.
  Locked.

Alternative considered + rejected: collapse the TopBar to wordmark+status
only on non-TUNER tabs, regaining ~88dp of body space. **Rejected**
because the resulting layout shift on tab switch is jarring (the bigger
problem than the lost body space). If body-space pressure becomes real on
DECK or SETUP, fix it at the body level (it's already a ScrollView on
SETUP, and DECK has the visual already pinned).

### 14.4 Status row placement — keep mic captions on every tab

Confirmed: **status row renders on all four tabs.**

- TUNER: obvious, needs the mic.
- METRO: in v1.2 the metronome may share the audio engine for TSF
  output; mic state is independent but the audio session lives in one
  place. If audio falls over (`stream-failed`), METRO output dies too —
  user deserves the heads-up.
- DECK: records from the mic. Status is critical here.
- SETUP: the user lands on SETUP **because** something went wrong with
  the mic and they want to change `hiFiMode` or restart. Hiding the
  status caption on the very tab where they came to fix it is hostile.

### 14.5 TABLE / PIPES placement — drop from non-TUNER tabs

Confirmed: TABLE and PIPES live only on the TUNER TopBar. Rationale in
14.2 (METRO section). If a user wants quick PIPES access from anywhere,
they switch to TUNER first — one extra tap, and the "PIPES belongs to
TUNER" mental model gets reinforced.

(Counter-argument: PIPES is a generic tuning-fork affordance, not strictly
TUNER-specific. Captured in 14.7 Q3.)

### Acceptance criteria additions (from Aragorn's edge-case audit)

These are **non-negotiable** even though they aren't questions per se;
they close real failure modes the council surfaced.

1. **`clickVolume × velocity` clamp**: result clamps to `[1, 127]`;
   `clickVolume === 0` is full mute (suppress the noteOn entirely, do
   not fire at velocity 0).
2. **TSF warm-up max-bar timeout**: WAV fallback is bounded; after N
   bars (default 4) with TSF still not ready, fire a `synth-
   unavailable` status event so the UI can show it. Don't run the
   fallback in perpetuity.
3. **`patternJson` deserialization**: AsyncStorage read returns a
   string that is `JSON.parse`'d; on any throw or schema-shape
   mismatch, silently reset to the default pattern. Do not crash the
   metronome on corrupted prefs.
4. **Live beat-flash boundaries**: no auto-scroll, no Animated, no
   per-beat useEffect; purely a render-time computation of
   `pulse % numerator === beatIndex`.

### R4 — Numerator + denominator picker UX in CUSTOM mode (refines §3 and §15.Q11.5)

User directive (verbatim): *"for that one a scroll up a list that goes
all the way up to 32 feels kind of s***** maybe we should just have up
down errors and a text entry field"*

The original spec phrase "scroll digit things" got drafted as a wheel-
picker in section 3; with the cap at 32 the wheel is too long to feel
good. Replace it.

**New numerator picker affordance:**

A horizontal row inside the CUSTOM-mode panel:

```
  [−]  [ 5 ]  [+]
```

- **`−` button**: decrement by 1 (clamp at 1).
- **`+` button**: increment by 1 (clamp at MAX_NUMERATOR = 32).
- **Center field**: a `TextInput` with `keyboardType="number-pad"`,
  `maxLength=2`, value bound to the current numerator. Tapping it
  raises the number-pad keyboard; the user types directly (1..32).
  On blur or submit, clamp + sync; on invalid input (e.g., 0 or 99),
  revert to the prior valid value without surfacing an error toast
  (silent guardrail, per the no-modal-on-every-error principle).

**Denominator picker** (same row, or a sibling row to the right):

```
  [−]  [ 8 ]  [+]
```

But denominator is constrained to standard musical values:
{2, 4, 8, 16, 32}. `−` and `+` walk that ladder; the text field
accepts only those five values (anything else snaps to the nearest on
blur).

Both controls reuse the existing `metroBpmFlankStepper` style (the BPM
± stepper pattern from MetroScreen) for visual consistency.

**Why arrows + text entry over a wheel:**

- Wheel pickers are interaction-heavy at high values (16 swipes to
  reach 32 from 1).
- Arrows give fine-grained adjustment when the user is dialing in
  +1/−1 changes (e.g., "I want 7/8, not 6/8").
- Direct entry gives O(1) input for non-adjacent jumps ("I want 13/8
  not 4/4").
- Pattern matches the BPM stepper in METRO and the A= stepper in
  TopBar — internal consistency across the app.
