# v1.3 METRO Redesign — Per-Tab Top Pills, User Profiles

**Author:** Frodo (design pass, no code edits)
**Status:** DRAFT — locked sections marked. Council/orchestrator decides §11
items before implementation.
**Audience:** Sauron (state + persistence), the React UI implementer, and the
follow-up Frodo review pass.

**Mirrors** the structure of `docs/v1.2-metronome-design.md`. Where v1.3

---

## 1. Goals (user spec, 1-paragraph restate)

v1.2 shipped metronome customization (CUSTOM time-sig + per-beat pattern +
subdivisions + drum picker) as a new `METRO VOICES` group in SETUP. The user
finds that SETUP is now cluttered with controls that only belong to METRO,
and wants per-tab settings to live on the tab they belong to. v1.3 moves all
metronome customization off SETUP onto an in-tab pill nav: METRONOME (main
controls) on the left, CUSTOMIZATION (per-profile editor) on the right.
Bottom-of-METRONOME gets a row of 8 pills — 4 preset time-sigs (2/4 3/4 4/4
6/8) and 4 saved **user profile slots** (User 1..4). Each profile bundles
time-sig + drum kit + beat pattern + subdivisions + a user-editable name, so
a player shedding an odd-meter etude for two weeks can recall their full
metronome setup in one tap. SETUP shrinks to globals only. TUNER and DECK
adopt the same top-pill pattern where it earns its keep — see §7.

User invariant (verbatim, supersedes any conflicting v1.2 council call):
> "move per tab things into top pill menus, like metronome customization
> should live in the metro settings. keep settings settings for globals"

## 2. Top-pill nav pattern (visual + interaction)

A new sub-navigation surface that sits **inside** a tab body, **below** the
TopBar (v1.2 §14, 136dp locked) and **above** the tab's content body. It is
NOT a second tab bar — it scopes only to the parent tab, and disappears (or
is re-rendered with that tab's pills) on tab switch.

### Visual

```
   ┌─ TopBar (locked 136dp, v1.2 §14) ──────────────────┐
   │  BELLCURVE              · WARMING UP               │
   │  [BADGE]  [A= 440]                                 │
   └─────────────────────────────────────────────────────┘
   ┌─ Sub-pill row (v1.3, NEW, fixed height) ───────────┐
   │     [ METRONOME ]    [ CUSTOMIZATION ]             │  ← 2 pills, equal width
   └─────────────────────────────────────────────────────┘
   ┌─ Sub-page body ────────────────────────────────────┐
   │                                                     │
   │   (either MetronomeSubPage or CustomizationSubPage) │
   │                                                     │
   └─────────────────────────────────────────────────────┘
   ┌─ TabBar (METRO highlighted) ────────────────────────┐
```

### Interaction

- Two pills, full-width row, equal flex, **fixed minHeight of 48dp** (touch
  target). Reuse the existing `gainPill` / `gainPillActive` styles already
  used for output route, filter mode, theme picker, etc. Do not invent a new
  pill style. Add ONE new wrapper style `subPillRow` that matches the
  existing `setupVoicePresetRow` pattern with equal-width children.
- Active pill = `gainPillActive` (filled accent). Inactive = `gainPill`
  (outlined). Tap an inactive pill swaps the body content; the row itself
  never moves or changes height.
- **Locked height across both sub-pages.** The sub-pill row reserves the
  same height whether METRONOME or CUSTOMIZATION is active (per v1.1
  anti-flicker rule + v1.2 §10). 48dp + 8dp top/bottom padding = 64dp total
  budget for the row.
- Sub-page state (`activeSubPill`) is **local component state** in the tab
  screen, NOT persisted to AsyncStorage. On tab change away and back, the
  user lands on METRONOME (the active-by-default sub-pill). Rationale: a
  player almost always wants the main controls, not the editor. (Captured
  in §11 Q3 if user wants persist-last-active behavior.)

### What this is NOT

- NOT a bottom sheet. The sub-page swap is in-place; the user is not
  ambushed by a slide-up modal.
- NOT a deep stack. There is no back arrow. Tapping the other pill returns
  the user; tapping the same pill is a no-op.
- NOT carrying transient state. If the user is mid-edit on CUSTOMIZATION
  (editing User 3's name) and swaps to METRONOME, the edit is committed on
  blur exactly as it would have been (see §4 editor commit rules).

## 3. METRONOME sub-page layout

This is the v1.2 MetroScreen body, with one structural change: the
time-signature pill row at the bottom expands from 5 pills to **8 pills** —
4 preset + 4 user-profile slots. The v1.2 `CUSTOM` pill is **gone**; its
job is now performed by tapping a User slot whose profile happens to use a
custom signature. The inline `CustomTimeSigPanel` that v1.2 rendered below
the row when CUSTOM was active is also gone from METRONOME — it relocates
to CUSTOMIZATION (§4).

### Wireframe

```
   [−5] [−1]   120   [+1] [+5]                          ← BPM row (unchanged)

   ┌─────────────────────────────────┐
   │  (PULSE / FLASH / PENDULUM)     │                  ← visualizer (unchanged)
   │              · ·                │
   │            · · · ·              │
   └─────────────────────────────────┘

              [   TAP   ]                                ← TAP (unchanged)
              [  START  ]                                ← START/STOP (unchanged)

   ┌── Bottom row (NEW shape — 8 pills, 2 rows of 4) ──┐
   │   [ 2/4 ] [ 3/4 ] [ 4/4 ] [ 6/8 ]                 │  ← presets
   │   [ ★1  ] [ ★2  ] [ ★3  ] [ ★4  ]                 │  ← user slots
   └────────────────────────────────────────────────────┘
```

- **Visualization still flex:1**, absorbs height changes from the bottom
  row growing 1→2 rows. The hard layout rule (no ScrollView on METRO) is
  preserved by stealing height from the visualization, exactly as v1.2 §10
  budgeted for the CUSTOM stepper.
- **Pill grid**: 2 rows × 4 columns. Reuse existing `metroSigPill` style;
  add `metroSigPillUser` modifier that has a slightly different accent so
  the user can see at a glance which pills are presets and which are user
  slots. (`borderStyle: 'dashed'` on idle; same `metroSigPillActive` when
  selected. The label inside makes the distinction obvious anyway.)

### User-slot pill content

Each user-slot pill shows its **shortened display name** (truncated
single-line). Defaults are `User 1`, `User 2`, `User 3`, `User 4` (see §9
for default profile content). The user renames slots from the CUSTOMIZATION
sub-page (§4). Truncate with `numberOfLines={1}` + `ellipsizeMode="tail"`;
fontSize 11; max width = the pill's natural flex width (~72dp on a
360dp-wide phone). Do NOT render the resolved time-sig "7/8" inside the
pill — the name is what the user assigned, and that's what they look for.

### Tapping behavior

- **Tap a preset pill** (2/4, 3/4, 4/4, 6/8): identical to v1.2 — calls
  `metro.setTimeSig({ kind: 'preset', value: '...' })`. Drum pattern is
  the **default kick+click** (since presets carry no profile). Subdivisions
  default OFF. Effectively: tapping a preset RESETS pattern+subdivisions
  back to defaults for that beat count. This is the implicit v1.2 behavior
  already (presets don't carry pattern data) and v1.3 keeps it.
- **Tap a user slot** (★1..★4): calls a new `metro.loadProfile(slotIdx)`
  which atomically restores timeSig + pattern + patternMode + subdivisions
  + subdivisionVoice from the slot. The metronome continues running
  through the swap (bar position resets to "1" same as setTimeSig — never
  glitches mid-beat). If the slot is empty (defaults), the user just hears
  the slot's default settings.
- **Long-press**: skip in v1.3. v1.2 §10 already bans long-press for
  non-discoverability. If the user wants to delete or rename a slot,
  they go to CUSTOMIZATION.

### Active pill highlight

The active user slot is highlighted with `metroSigPillActive` even though
the underlying timeSig may "happen to" equal a preset. Rationale: the
user picked a SLOT, not a TIME SIG. Even if User 1 is set to 4/4, the 4/4
preset pill should NOT light up — instead, the User 1 slot lights up. This
is a "selection source" tracking concern, not a "currently-resolved
timesig" concern; see §8 state-machine extensions.

Edge case: what if the user taps a preset while a user slot is active?
The preset becomes active (slot deselects). The user's profile slot data
is NOT mutated by this — it stays as-is in storage, ready to be re-loaded
next time the slot is tapped. This matches the v1.2 "Don't reset the
user's custom pattern when they toggle DEFAULT KIT back on" invariant.

### Removed from METRONOME sub-page

- The v1.2 inline `CustomTimeSigPanel` — gone; moves to CUSTOMIZATION
  per-profile editor (§4).
- The v1.2 `CUSTOM` pill — gone; user slots replace it.
- The v1.2 `metro2_customSpacer` reserved-height block — gone. The
  bottom row's height is fixed at "2 rows of pills" always, so there is
  no need for a reserve-spacer.

## 4. CUSTOMIZATION sub-page layout

Settings-style editor for the 4 user profile slots. **This sub-page MAY
scroll** — it's the editor view, not the in-session view. v1.2 §10's
"no ScrollView on METRO main" rule applies only to the MetroScreen body
itself; this sub-page renders inside the same tab but is a different
component, so a ScrollView here is fine.

### Wireframe (CUSTOMIZATION active)

```
   ┌─ Sub-pill row ─────────────────────────────────────┐
   │     [ METRONOME ]    [▣ CUSTOMIZATION ]            │
   └─────────────────────────────────────────────────────┘
   ┌─ Sub-page body (ScrollView) ───────────────────────┐
   │  CUSTOM SOUND SET · CUSTOM BEAT PATTERNS           │  ← header label (§4 naming)
   │                                                     │
   │  ─────── USER 1 ───────                             │
   │  Name:    [ Brubeck 5/4 etude       ]               │
   │  Time signature:                                    │
   │    [ Preset ] [ Custom ]                            │
   │    Preset:  [2/4][3/4][4/4][6/8]                    │
   │    Custom:  num [−][ 5 ][+]   den [−][4][+]         │
   │  Beat pattern:                                      │
   │    [PerBeatRow — same as v1.2]                      │
   │  Subdivisions:                                      │
   │    [OFF][8TH][16TH][TRIPLET]                        │
   │    Sub voice: Closed Hi-Hat (042)  [tap → picker]   │
   │                                                     │
   │  ─────── USER 2 ───────                             │
   │  ...                                                │
   │                                                     │
   │  ─────── USER 3 ───────                             │
   │  ...                                                │
   │                                                     │
   │  ─────── USER 4 ───────                             │
   │  ...                                                │
   └─────────────────────────────────────────────────────┘
```

### Two display modes for the list

**Default mode** = all 4 slots collapsed to header rows. Tap a header to
expand a single slot's editor. Only one slot is expanded at a time
(accordion). Rationale: showing 4 full editors stacked is overwhelming and
forces a long scroll on a 720dp phone. An accordion shows the user the
list of slots (so the slot they want is one tap away) and the editor for
the active slot (so they can change one thing in one place).

Collapsed header row content:
```
   USER 1 · "Brubeck 5/4 etude"      5/4  ·  HiHat 8ths       ▶
```
Tap → expands; tap an open header → collapses; tap another → that one
expands and the previously-open one collapses. Smooth in-flow, no
animation gimmickry; same as the v1.2 `setupVoiceMoreToggle` pattern.

(Captured in §11 Q1 — alternative: render all 4 slots inline, no
accordion. Frodo recommends accordion. Decision needed.)

### Per-slot editor contents (when expanded)

Listed top-to-bottom in the editor for one slot:

1. **Name** — single-line `TextInput`. `maxLength` = 24 chars (fits the
   pill back on METRONOME without truncating most names). Placeholder =
   slot's default name (`User 1` etc). Commit on blur (no save button
   needed; AsyncStorage write is debounced 500ms exactly as the v1.2
   prefs save already debounces). On blur with empty string → fall back
   to default name. Reuse a TextInput style that matches the existing
   `metroBpmStepperText` font weight + size; new style key
   `customizationNameInput`.
2. **Time signature** — a **kind toggle** (`[Preset] [Custom]` two-pill,
   same `gainPill` idiom). When `Preset` is active, render the 4 preset
   pills row (2/4 3/4 4/4 6/8). When `Custom` is active, render the
   existing `CustomTimeSigPanel` (numerator/denominator steppers + text
   entry per v1.2 R4). The kind toggle is itself a per-profile setting,
   stored in the slot.
3. **Beat pattern** — `PerBeatRow` exactly as v1.2 (with horizontal
   scroll when N > 8). The drum picker bottom sheet is hosted at the
   tab-screen level (same lifting pattern as v1.2 SetupScreen). The
   picker target now carries an additional `profileSlot: number` field
   so on selection the write goes into `profiles[slot].pattern[idx]`
   not the live metronome's pattern. **The previously-edited slot is
   also the live slot** if the user has it loaded — see §8 sync rules.
4. **Subdivisions** — radio row `[OFF][8TH][16TH][TRIPLET]` (the v1.2
   `SubdivisionPicker` component used as-is, with the sub-voice picker
   row underneath it as the v1.2 design already shows).
5. **(Deferred to §11 Q2)** Master volume offset for this profile (e.g.
   "User 4's salsa pattern is too loud, dial it down 6 dB"). Frodo
   recommends DEFER — `clickVolume` is global and the user spec doesn't
   ask. Open question.

### Header label naming (per user spec)

User said *"rename the profile set custom sound set, custom beat patterns
and so on."* — the sub-page top header reads:

> **CUSTOM SOUND SET · CUSTOM BEAT PATTERNS**

It's a labeled section header (single line, `settingsGroupLabel` style),
not a button — it just orients the user. Below it, each slot header
("USER 1") is the actionable affordance.

Each section label INSIDE a profile editor uses plain descriptive labels
(see wireframe): `Name`, `Time signature`, `Beat pattern`, `Subdivisions`.
Sentence-case is fine since these are field labels, not nav chrome.

### Save / commit semantics

- All edits commit **live** (no Save button, no Discard button).
  AsyncStorage writes are debounced as v1.2 prefs already debounce.
- Edit-while-loaded sync: if the user has User 2 currently loaded and is
  editing User 2's pattern, the live metronome's pattern updates on
  every tap of the per-beat editor (so the user hears the change next
  beat). If they're editing a slot that ISN'T currently loaded, the
  changes go only to the slot in storage; the live metronome is
  unaffected.
- Renaming a slot that's currently loaded does NOT change the live
  state — only the slot label. The METRONOME pill row's label updates
  because it reads slot names from storage.

### Delete / reset a slot?

- **No delete** (slots are fixed-cardinality = 4). Captured in §11.
- **Reset to default**: a small `Reset` text button at the bottom of the
  expanded editor: `Reset this slot to default`. Tap → confirms inline
  (a 2-button inline confirm using the v1.2 `metroSigPill` row idiom:
  "Reset?" / "Cancel"). Restores the slot to its fresh-install state
  (§9).

### Drum picker host

Lives on the CustomizationSubPage component (not SetupScreen anymore).
Picker target type extends:

```ts
type PickerTarget =
  | { kind: 'beat'; slot: number; index: number }
  | { kind: 'sub'; slot: number }
  | null;
```

The sheet itself (`DrumPicker` component) is reused unchanged. Only the
target shape and the callback that writes the result change.

## 5. Profile slot data model

### State shape (new types)

```ts
// One profile slot. Captures the v1.2 customization surface plus a
// user-editable name.
export interface MetroProfile {
  /** User-editable display name. Defaults to 'User 1'..'User 4'. Empty
   *  string on read coerces to the default. */
  name: string;
  /** Time signature — same tagged union as v1.2's TimeSig. */
  timeSig: TimeSig;
  /** Per-beat pattern. Length === beatsPerBar(timeSig); coerced on load. */
  pattern: BeatInstrument[];
  /** Pattern mode. v1.2 carries this; v1.3 preserves it per-profile. */
  patternMode: 'default' | 'custom';
  /** Subdivision setting. */
  subdivisions: Subdivision;
  /** Sub-tick voice for THIS profile. */
  subdivisionVoice: BeatInstrument;
}

/** Exactly four profile slots. Index 0..3 == User 1..4. */
export type MetroProfileSlots = [
  MetroProfile, MetroProfile, MetroProfile, MetroProfile,
];
```

### Persistence

Stored as a single JSON-stringified field in `AppPrefs`:

```ts
// In AppPrefs:
/** v1.3 — JSON-stringified MetroProfileSlots (length=4). Bad JSON, bad
 *  length, bad inner shape → silently reset to defaults (§9), mirroring
 *  the v1.2 metroPatternJson tolerant-coercion guard. */
metroProfilesJson: string;
```

Single field over four fields keeps `AppPrefs` flat-ish and matches the
v1.2 `metroPatternJson` precedent. `coerceProfilesJson` lives next to the
existing `coercePatternJson`.

### Defaults

See §9. Fresh install: four profiles populated with sensible starter
values, distinct enough that the user can audition each.

### Active slot index

A new `AppPrefs` field tracks which slot (if any) is currently loaded
into the live metronome:

```ts
/** v1.3 — currently loaded profile slot (0..3) or null if none (the user
 *  is on a preset, or hasn't tapped a User slot since launch). */
metroActiveProfileSlot: number | null;
```

Persisted so that closing and re-opening the app restores the slot the
user was using. On a load (`metro.loadProfile(slot)`) this flips to the
new index. On a preset tap, this flips to null.

## 8. State machine extensions to useMetronome

### New API surface

```ts
export interface MetronomeState {
  // --- v1.1 / v1.2 unchanged ---
  bpm, setBpm, bumpBpm,
  running, start, stop, toggle, registerTap,
  beat, pulse,
  clickVolume, setClickVolume,
  timeSig, setTimeSig,
  pattern, setBeatInstrument,
  patternMode, setPatternMode,
  customNum, customDen, setCustomNum, setCustomDen,
  subdivisions, setSubdivisions,
  subdivisionVoice, setSubdivisionVoice,
  kit,

  // --- v1.3 ---
  /** The 4 profile slots. Stable identity across renders; mutations go
   *  through updateProfile/loadProfile/resetProfile. */
  profiles: MetroProfileSlots;
  /** Active slot index 0..3, or null if a preset is the active source. */
  activeProfileSlot: number | null;
  /** Atomically load slot's full settings into live state. Resets bar
   *  position the same way setTimeSig already does. */
  loadProfile: (slot: number) => void;
  /** Edit one field of a slot. If the slot is the active slot, mirror the
   *  change into live state so the user hears it next beat. If not, only
   *  the stored slot updates. */
  updateProfile: (slot: number, patch: Partial<MetroProfile>) => void;
  /** Rename a slot — convenience over updateProfile(s, { name }). */
  renameProfile: (slot: number, name: string) => void;
  /** Restore a slot to its fresh-install defaults (§9). Does NOT change
   *  activeProfileSlot unless `slot === activeProfileSlot`; if it is, the
   *  live state is reloaded from the now-default slot. */
  resetProfile: (slot: number) => void;
}
```

### Active-slot tracking — why a separate flag

A naive implementation would compare live `timeSig` / `pattern` /
`subdivisions` to each slot and infer "you're on slot 2." That **breaks
the moment the user touches anything**: if they have User 2 loaded and
nudge subdivisions from OFF to 8TH inline (not via CUSTOMIZATION),
they're no longer "on slot 2" by value, but they obviously ARE still on
slot 2 by selection-source.

Tracking `activeProfileSlot` explicitly is therefore necessary. The flag
flips on the events that explicitly designate a selection source:
- `loadProfile(s)` → `activeProfileSlot = s`.
- `setTimeSig({kind:'preset',...})` triggered by a preset pill tap →
  `activeProfileSlot = null`.
- `setTimeSig({kind:'custom',...})` triggered programmatically by
  `loadProfile` does NOT flip the flag (loadProfile sets it directly).

### The "fifth pseudo-time-sig kind" question

The user-directive shape suggests treating user profiles as a third kind
of TimeSig (alongside `preset` and `custom`). Frodo's call: **NO.** The
TimeSig type stays as the v1.2 tagged union; the "selection source"
concern is handled by `activeProfileSlot` (a sibling field), not by
extending `TimeSig`. Reasons:

- A user profile's `timeSig` is **itself** either a preset or a custom
  — extending the union to add `{ kind: 'profile'; slot: number; ... }`
  would require expanding it into preset/custom on every read.
- The `TimeSig` type is widely consumed (display helpers, scheduler,
  prefs migration). Extending it is invasive.
- The selection-source concern is a UI presentation concern (which pill
  to highlight), not a scheduling/audio concern. Keep it in UI state.

So: tapping a User slot calls `loadProfile(s)`, which internally calls
`setTimeSig(profile.timeSig)` (preset or custom) plus the other field
setters, plus sets `activeProfileSlot = s`. The METRONOME pill row reads
`activeProfileSlot` to decide which pill lights up. If it's null, the
row falls back to highlighting based on `timeSig`.

### Edit-while-loaded sync

When `activeProfileSlot === s` and the CUSTOMIZATION sub-page calls
`updateProfile(s, { subdivisions: '8th' })`, the hook:

1. Writes `profiles[s].subdivisions = '8th'` (storage).
2. Also calls `setSubdivisions('8th')` (live state).

This way the user hears the edit immediately. If the slot being edited
is NOT active, only step 1 runs.

### Scheduler — unchanged

The scheduler reads from live state (`timeSig`, `pattern`,
`subdivisions`, `subdivisionVoice`) exactly as v1.2. It does not know or
care about profiles — they're a UI/state-management concept layered on
top. This is intentional: it keeps the audio path simple.

### AppState background pause — unchanged

The `wasRunningRef` pattern from v1.2 §10 still works. Profiles add no
new lifecycle concern.

## 10. Migration from v1.2 prefs

A v1.2 user has these prefs populated:
- `metroTimeSigKind`, `metroTimeSigPreset`, `metroCustomNumerator`,
  `metroCustomDenominator`.
- `metroPatternJson`.
- `metroSubdivisions`, `metroSubdivisionVoiceMidi`,
  `metroSubdivisionVoiceVelocity`.

Plus a derived `patternMode` (v1.2 used `'default' | 'custom'` per the
v1.2 §7 type but it's not clear if it landed in AppPrefs; verify against
current prefs.ts — if not stored, default to `'custom'` when migrating
since a v1.2 user who edited their pattern at all has implicit custom
mode).

### Migration logic in `loadPrefs`

1. If `metroProfilesJson` is missing or fails `coerceProfilesJson`:
   - **Build the v1.2 user's settings into User 1**. So a v1.2 user's
     work isn't lost: the slot they were effectively using becomes User 1.
   - User 2, User 3, User 4 get the fresh-install defaults from §9
     (Jazz waltz / Odd meter / Blank slate).
   - Set `metroActiveProfileSlot = null` initially (the v1.2 user wasn't
     using a slot; preserve their preset/custom feel until they tap a
     slot).
2. The legacy v1.2 fields (`metroTimeSigKind` etc.) stay in `AppPrefs`
   for one minor release (v1.3) — they remain the single source of
   truth for the live metronome on a fresh launch (v1.3 only stores
   profile slots, not active live state). Live state seeds from those
   legacy fields on mount, exactly as v1.2 did. On v1.4 we may consolidate.

   Alternative: live state derives from `metroActiveProfileSlot` →
   `profiles[slot]` if slot is non-null, else from the legacy fields.
   Frodo prefers this — see §11 Q7.

### What v1.3 writes back

On every prefs save:
- Write `metroProfilesJson` (always).
- Write `metroActiveProfileSlot` (always).
- Continue writing the legacy v1.2 fields (`metroTimeSigKind`,
  `metroTimeSigPreset`, `metroPatternJson`, etc.) so a user who
  downgrades from v1.3 → v1.2 doesn't lose their live settings — same
  one-release backward-compat policy v1.2 applied for legacy `timeSig`.
  Drop in v1.4.

### Corrupt or partial JSON

`coerceProfilesJson` follows the same defensive-parse pattern as
`coercePatternJson`:
- Bad JSON → null → defaults.
- Non-array or length ≠ 4 → null → defaults.
- Any element with bad shape (missing name, bad timeSig kind, pattern
  array fails inner coercion, subdivisions not in the allowed set) →
  drop ONLY that element to its default, keep the others. (Different
  from `coercePatternJson` which all-or-nothing; here we want partial
  recovery because losing 3 slots due to 1 corrupt is harsher than
  losing 1.)

## 18. MIDI alignment (v1.3 addition)

User directive (verbatim): *"state machine needs to be way more midi
friendly obv"*. Gandalf is folding a `useMidiBus` layer into his
state-machine scrub plan. This section restates §5 (data model) and
§8 (state machine) in MIDI-shaped terms and rewires §3 (live beat
flash) to subscribe to the bus instead of polling `pulse`. **Where
§18 disagrees with §5 / §8 / §11.Q12, §18 wins.**

### 18.1 Profile data model — MIDI-shaped

Each user profile is fundamentally a **named bundle of MIDI channel
state plus a tempo-grid shape**. v1.3 only exposes one channel
(`drums`, GM channel 10 / zero-indexed 9), but the model is open so
v1.4 can add more channels (e.g., a `aux` cowbell channel that also
plays a pattern alongside drums).

```ts
/** v1.3 MIDI-aligned profile shape. Replaces §5's MetroProfile. */
export interface MetroProfile {
  /** User-editable display name (was §5). */
  name: string;
  /** Bar shape — beats + denominator. Channels schedule against this. */
  timeSig: TimeSig;
  /** Per-channel state. v1.3 ships only `drums`; type allows more. */
  channels: {
    drums: DrumChannelState;
    // v1.4+: aux?: AuxChannelState; bass?: BassChannelState; ...
  };
  /** Per-profile master gain (0..1). v1.3: stored but no UI surface
   *  (per §11.Q2). Multiplied into noteOn velocity at the bus. */
  masterGain: number;
}

/** Drum channel state for one profile. Maps cleanly to a MIDI
 *  channel-10 setup: a "program" (the drum kit/bank choice), a
 *  per-beat note schedule, and a sub-tick voice + count. */
export interface DrumChannelState {
  /** Currently always 'gm-standard' in v1.3 — TSF GM channel 10
   *  ignores Program Change on the drum channel, so this is forward-
   *  compat only (e.g., 'gm-room', 'gm-electronic' if we add SF2
   *  banks later). Stored as a string id so the open set is open. */
  program: string;
  /** Per-beat noteOn schedule. Length === beatsPerBar(timeSig). */
  pattern: BeatInstrument[];
  /** Pattern mode — preserved from v1.2. */
  patternMode: 'default' | 'custom';
  /** Sub-tick rate. */
  subdivisions: Subdivision;
  /** Sub-tick voice (same shape as a beat instrument: midi note +
   *  velocity on channel 10). */
  subVoice: BeatInstrument;
}
```

The shape change vs §5 is structural: pattern/subdivisions/subVoice/
patternMode now live **inside `channels.drums`**, not at the profile
root. Persistence still serializes to `metroProfilesJson` as one blob
(§5.persistence unchanged). The migration coerce (§10) reshapes a
v1.2 user's settings into `channels.drums` instead of profile-root
fields.

### 18.2 User-facing language — friendly, MIDI internals hidden

Per §17.R3 / §16.U13 (simple by default, complex behind toggle), the
CUSTOMIZATION sub-page **does not** show channel numbers or program
selectors by default. UI labels:

| Internal (MIDI) | UI label |
|---|---|
| `channels.drums.program` | "Drum kit" |
| `channels.drums.pattern[i]` | "Beat N" (the per-beat cell) |
| `channels.drums.subdivisions` | "Subdivisions" |
| `channels.drums.subVoice` | "Sub voice" |
| `masterGain` | (no UI in v1.3) |

A future "Show advanced" toggle (§11.Q-new — see 18.6) could expose
the channel + program selectors directly, for users who want to wire
their own banks. v1.3 hides it.

The CUSTOMIZATION sub-page section header from §4 is unchanged:
`CUSTOM SOUND SET · CUSTOM BEAT PATTERNS` (the user explicitly
requested this phrasing in §1; "sound set" is the user-friendly term
for "kit + program" and "beat patterns" for "the per-channel note
schedule").

### 18.3 `useMidiBus` boundary — what useMetronome owns vs what the bus owns

This is the load-bearing decoupling. Gandalf's bus owns:

- **Channel ownership**: which feature (metronome, drone, pitch
  pipes) is currently driving which MIDI channel. v1.3 metronome
  claims channel 10 (drums); future polyphonic features negotiate.
- **noteOn / noteOff dispatch**: the actual TSF synth call goes
  through `bus.noteOn(channel, midi, velocity)` instead of the
  v1.2 `synth.noteOn(...)` direct call. This lets the bus apply
  gain, mute, channel-claim checks, and emit observable events.
- **High-level event stream**: `bus.on('noteOn', listener)` fires
  for every noteOn dispatched on any channel. UI components
  subscribe.

useMetronome owns:

- The **schedule** (wall-clock targets per beat, sub-tick offsets,
  AppState pause/resume).
- The **profile state** (slots, active slot, edit-while-loaded
  sync from §8).
- Resolving a scheduled beat into a `bus.noteOn(channel='drums',
  midi, velocity*masterGain*clickVolume)` call.

The bus replaces the v1.2 direct-to-`synth` calls. WAV fallback
(v1.2 §15.Q11.7) flows through the bus too — the bus has its own
fallback emitter so the WAV path is just another channel
implementation under the same observable surface.

### 18.4 Live beat flash — subscribes to the bus, not `pulse`

v1.2 §15.Q11.12 spec'd the PerBeatRow's live-cell highlight as
`pulse % numerator === beatIndex`, computed at render time. v1.3
**moves this to a bus subscription**:

```ts
// Inside PerBeatRow (or its caller in CustomizationSubPage):
const [liveBeatIdx, setLiveBeatIdx] = useState<number | null>(null);
useEffect(() => {
  const off = bus.on('noteOn', (evt) => {
    if (evt.channel !== 'drums') return;
    if (evt.tick !== 'beat') return; // ignore sub-ticks for flash
    setLiveBeatIdx(evt.beatIndex);
  });
  return off;
}, [bus]);
```

This satisfies Gandalf's decoupling goal (UI no longer reads
scheduler internals) while preserving the v1.2 acceptance criteria
(no Animated, no auto-scroll, accent-via-borderColor only — those
constraints apply equally to a subscriber-driven render).

**Aragorn-equivalent guard**: the bus event MUST fire on the JS
thread synchronously at noteOn-time (not deferred to a microtask),
or the flash drifts behind the audio. If the bus is implemented
with an async queue, an additional `beat` event channel may be
needed; Gandalf's design pass settles this.

### 18.5 Picker target carries channel — minor change to §4

The §4 drum picker `PickerTarget` extends to identify the channel,
not just the slot:

```ts
type PickerTarget =
  | { kind: 'beat'; slot: number; channel: 'drums'; index: number }
  | { kind: 'sub';  slot: number; channel: 'drums' }
  | null;
```

v1.3 only ever sets `channel: 'drums'`, but the field exists so
v1.4 can route a per-beat picker to a different channel without
breaking the target shape.
