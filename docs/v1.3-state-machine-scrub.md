**Author:** Gandalf the Grey (architecture pass, no code edits)
**Status:** DRAFT — needs user sign-off on the Open Questions section (§8)
before implementation begins.
**Audience:** the implementation agent(s) executing the per-commit sequence in

## 0. Reading guide

This is the architectural counterweight to v1.0's war-council finding on the
`useAudioEngine.ts` god-hook. If you implement, read §2 (the split), §4
(`useUiPrefsStore` contract — the most disruptive cut), §6 (commit sequence),
and §7 (the things that look splittable but aren't). If you review, read §2
and §7 first.

The v1.2.1 hotfix (in flight in a sibling agent) wraps the four big hooks'
returns in `useMemo`. That kills the render cascade. **It is a band-aid.**
The cascade keeps happening *inside* `useAudioEngine` itself — every audio
buffer churns ~10 setStates against a 50-field struct, every consumer that
reads ANY of those fields re-renders. The real fix is to make the audio-rate
state, the user-rate state, the persisted-pref state, and the COLLECT-mode
state into separate hooks with separate identities, so the consumers that
only need user-rate state are not chained to the meter.

### 1.2 useMetronome — 901 lines, mostly justified

Counts: **9 useState**, **5 useEffect**, **18 useCallback**, **15 useRef**.

Return surface (`MetronomeState`, §143-186 of `useMetronome.ts`): **22 fields**.
Post-v1.2.1: returned object is `useMemo`-wrapped (line 854). All setters
are `useCallback`-wrapped. The hook is doing one large job (scheduler +
TSF + WAV fallback + tap-tempo + lifecycle + prefs persistence) but
unlike useAudioEngine, the job has a single coherent center of gravity:
the wall-clock scheduler. Splitting would invite coordination bugs.

One legitimate split target: `clickVolume` is unrelated to the scheduler
(it's a single user-rate value with its own persistence path, §731-742).
Could live in `useUiPrefsStore` (§4) since it's just a slider that gets
multiplied through to noteOn velocity at audio dispatch. Keep the existing
clickVolumeRef inside `useMetronome` for the dispatch read; let the slider
binding move out.

Audio-rate churn: `pulse` and `beat` change at beat rate (0.5–5 Hz for
30–300 BPM). Far below the audio engine's 40 Hz. Acceptable to leave them
in the metronome's struct as long as the struct is memoized — which it is
post-v1.2.1.

Counts: **3 useState**, **4 useEffect**, **14 useCallback**, **1 useRef**.

### 1.4 useDrone — 416 lines, also healthy

Counts: **1 useState**, **8 useEffect**, **8 useCallback**, **3 useRef**.

## 2. Proposed split for useAudioEngine

Five sibling hooks, composed top-down by a new thin coordinator (`useAudio`).
The composition is one-way: each downstream hook depends on the previous
one's outputs (or sibling refs), no cycles.

```
App.tsx
  └── useAudio()  (thin composition hook — see §2.6)
        ├── useUiPrefsStore()        — § 2.1, persisted user prefs (NO audio)
        ├── useAudioStream(prefs)    — § 2.2, capture + watchdog + retry
        ├── usePitchPipeline(stream, prefs) — § 2.3, YIN + filter + vote + drone duck
        ├── useCollectBuckets(pitch, prefs) — § 2.4, COLLECT + quality gate + session
        └── useDroneChaseWiring()    — § 2.5, drone-chase refs (or fold into 2.3)
```

The five-way split (plus the coordinator) corresponds 1:1 to clusters A–G of
§1.1, with E pulled out as the load-bearing cleanup.

### 2.1 useUiPrefsStore

**Pure UI/settings state.** No audio, no refs, no effects beyond load/save.
Designed so that re-renders here only land on screens that *read* prefs
(SetupScreen does; TunerScreen reads `tunerStyle`; MetroScreen reads
`metroStyle`/`outputRoute`/`clickOffsetMs`; DeckScreen reads `deckStyle`).
Pulling these 17 fields out of `useAudioEngine` means an audio buffer no
longer cascades into SetupScreen.

**Owned state** (17 + `clickVolume` from useMetronome = 18):

```ts
{
  // Display + ergonomics
  theme,             // 'dark' | 'night' | 'light'
  nightDarken,       // 0.4..1.0
  nightWarmth,       // -1..1
  displayMode,       // 'griff' | 'klingend'

  // Capture knobs that aren't part of the stream wiring
  gainMode,          // 'low' | 'high'
  filterMode,        // 'fast' | 'normal' | 'slow'   ← see §7 for why this LIVES here
                     //   (pipeline reads it via ref, mode change resets filter state)
  hiFiMode,          // boolean
  lowCutDb,          // -80..-10
  peakLock,          // boolean (LIVE vs COLLECT)
  allowOutOfRange,   // boolean
  nickname,          // string
  instrumentKey,     // string

  // Visualization styles
  tunerStyle,        // 'arc' | 'strobe' | 'led'
  metroStyle,        // 'pulse' | 'pendulum' | 'flash'
  deckStyle,         // 'reels' | 'vu' | 'waveform'

  // Metronome calibration (NOT scheduler state — those stay in useMetronome)
  metroClickOffsetMs,    // -50..50
  metroOutputRoute,      // 'speaker' | 'wired' | 'bluetooth'
  metroClickVolume,      // 0..1 (moved IN from useMetronome.clickVolume)

  // Drone prefs (moved IN from App.tsx)
  droneVoice,            // string voice id
  droneVolume,           // 0..1
  droneSemitones,        // -12..12

  // Reference pitch (moved IN from App.tsx)
  refHz,                 // 430..450

  // Debug flag
  showDebugOverlay,      // boolean (moved IN from App.tsx)

  // Min-N for table (moved IN from App.tsx)
  minNVisible,           // number

  // Hydration sentinel
  prefsLoaded,           // boolean — replaces engine.prefsLoaded
}
```

**Owned setters**: one per field. All `useCallback`-wrapped with `[]` deps.
Each writes-through to AsyncStorage via a single shared debounce (§4.3).

**Owned effects**:
- One hydration effect (`loadPrefs()` on mount, sets all fields, flips
  `prefsLoaded` true). Replaces the current §1098-1147 block in useAudioEngine.
- One AppState 'background' flush effect (writes any debounce-pending values
  immediately). Replaces the App.tsx-side AppState listener at §304-317 AND
  the useAudioEngine listener at §1184-1194 — collapses to a single
  authoritative writer.

**Owned refs**: a single `pendingSaveRef` for the debounce.

**Consumes from siblings**: nothing.

**Identity stability**: returned object memoized. The dep list is ~25
scalars + 25 setters — but because all setters are stable and the scalars
only change on user touch, the returned object identity only changes when
the user actually changes a pref. Sub-second cascades cease.

### 2.2 useAudioStream

**Audio capture + retry + watchdog. No pitch logic.** Owns the bridge into
`useAudioStream`/`useRawAudioInput`.

**Owned state**:
- `status: EngineStatus`
- `streamErrorReason: string | null`
- `hiFiActive: boolean`
- `audioSourceLabel: string`

**Owned setters / actions**: `retryPermission`, `retryStream`.

**Owned effects**:
- Permission request (§1208-1220 of current useAudioEngine)
- Stream open/close (§1247-1353)
- isStreaming/audioSourceLabel sync (§1358-1414)
- Buffer-flow watchdog (§1378-1398)

**Owned refs**: `lastBufferAtMs`, `stopping`.

**Consumes from siblings**: `hiFiMode` (from `useUiPrefsStore`).

**Output**: a stable `onBufferSubscribe(handler)` callback OR a `bufferStream`
object (whichever the pipeline hook wants). The handler signature stays
`(buffer: { data: ArrayBuffer|Uint8Array; sampleRate: number }) => void`,
identical to today's `onBuffer` callback in useAudioEngine.

**Why this split is safe**: the stream hook does not need to know about
YIN, filters, or display state. The pitch pipeline does not need to know
which underlying expo-audio or raw-audio-input is active — it just gets
buffers.

### 2.3 usePitchPipeline

**The hot path.** YIN, filter, octave-jump guard, ring buffer, display
tick, incumbent voting, drone-chase suspicion logic.

**Owned state** (audio-rate; these are the cascade sources):
- `freqHz: number | null`
- `rmsDb: number`
- `meterFill: number`
- `incumbentMidi: number | null`
- `rawFreqHz: number | null`
- `yinCallCount: number`
- `micSilenced: boolean`

**Owned setters**: none directly (the actions are internal — buffer arrives,
state updates). Drone-chase wiring exported: `setDroneCurrentMidi`,
`installDroneDuckHandler`.

**Owned effects**:
- The `onBuffer` callback subscription (delegates the actual capture to
  `useAudioStream`).
- The RAF display-tick effect (§1736-1958).

**Owned refs**: ring buffer, analysisBlock, filterState, lastStablePitch,
silentBufferCount, displayFreqRing, displayRmsRing, displayRingHead,
displayRingCount, bufferDurationMsRef, incumbentMidiRef,
incumbentMidiPublishedRef, voteCountRef, winnerFreqsRef,
droneCurrentMidiRef, droneRequestDuckRef, droneSuspicionFramesRef,
droneLastDuckMsRef, qualityHistoryRef (shared with §2.4 — see "shared
refs" below).

**Consumes from siblings**:
- `useAudioStream`: the buffer stream / onBuffer subscribe.
- `useUiPrefsStore`: `gainMode`, `filterMode`, `lowCutDb`, `peakLock`,
  `instrumentKey` (read via ref pattern — same as today).

**Shared refs** with useCollectBuckets: `qualityHistoryRef`. The gate's
history struct is mutated by both the audio-rate gate (here) and
`logCurrentReading` (in §2.4). Pass the ref down from a parent (the
coordinator in §2.6) so both hooks see the same struct. **This is the
one shared-mutable concession in the split.** Alternative: lift the gate
into the pipeline and have §2.4 read decisions via a callback. See §7 for
why the ref-sharing is preferred.

**Why YIN+filter+vote+display all live in one hook**: see §7. The ring
buffer, sample rate, filter state, and display-tick window math are
coupled by the buffer cadence invariant — splitting them risks
coordinate-system bugs that won't show up in normal use but will surface
on a sample-rate or buffer-duration change.

### 2.4 useCollectBuckets

**COLLECT-mode accumulation + quality gate user-side + session auto-log.**

**Owned state**:
- `activeBucket: BucketStats | null`
- `sessionActive: boolean`
- `sessionStartedAtMs: number | null`
- `droppedFrameCount: number`
- `lastDropReason: QualityRejection`

**Owned setters / actions**: `clearActiveBucket`, `logCurrentReading`,
`undoLastLog`, `setSessionActive`.

**Owned effects**: none new — the active-bucket pump currently runs inside
the RAF tick (§1933-1948); move it into a dedicated RAF in this hook OR
have §2.3's RAF call a thin `pump()` exposed by this hook. The pump option
is cleaner (one RAF in the app, not two competing for the frame).

**Owned refs**: `bucketAccumsRef`, `activeBucketKeyRef`,
`lastEmittedBucketSig`, `sessionLastMidiRef`, `sessionStreakStartMsRef`,
`sessionLastAutoLogMsRef`, `droppedFrameCountRef`.

**Shared refs**: `qualityHistoryRef` (with §2.3).

**Consumes from siblings**:
- `usePitchPipeline`: needs `freqHz`/`rawFreqHz`/`rmsDb` (via refs) and the
  per-frame `QualityDecision` from the gate. Concretely: the pipeline
  exposes a `lastFrameRef` containing the most-recent buffer's nextRaw,
  exactMidi, roundedMidi, centsFromRound, confidence, gate decision, ms;
  this hook reads it inside its pump.
- `useUiPrefsStore`: `peakLock`, `instrumentKey`, A4 (`refHz`).

**Why split this from §2.3**: COLLECT/session logic owns its own state,
its own refs, its own setters, and is entirely independent of the YIN/vote
loop except for "what's the current frame's pitch+gate-decision." That's
the cleanest abstraction boundary in the file.

### 2.5 useDroneChaseWiring — OR fold into §2.3

Two options:

**(a) Standalone hook**: owns `droneCurrentMidiRef`, `droneRequestDuckRef`,
`droneSuspicionFramesRef`, `droneLastDuckMsRef`. Exports
`setDroneCurrentMidi`, `installDroneDuckHandler`. The pipeline reads the
refs.

**(b) Fold into `usePitchPipeline`**: drone-chase IS pitch-pipeline
concern. The suspicion counter increments inside the RAF tick using the
vote map. Separating it adds a hook-boundary for ~50 lines of code.

**Gandalf's call: (b).** Fold drone-chase into `usePitchPipeline`. The
benefit of (a) is "one more crisp hook"; the cost is plumbing four refs
across a boundary that's purely internal. See open question §8.Q1.

### 2.6 useAudio coordinator

A thin composition hook in `src/useAudio.ts` that composes the four (or
five) sibling hooks and returns a single object. Existing call sites
(`App.tsx` does `useAudioEngine()`) can switch to `useAudio()` and read
sub-fields — OR we keep `useAudioEngine` as a backward-compatible thin
wrapper that re-aggregates the split hooks into the original surface for
a transition period. See §5 (migration).

**Coordinator owns**: nothing (no state, no effects). It just calls the
four sub-hooks in order and returns `{ prefs, stream, pitch, collect }`
(or whatever shape the user picks — §8.Q2).

**Why a coordinator**: lets the screens import individual sub-hooks
without each screen having to call all four. SetupScreen wants
`{ prefs, stream }`. TunerScreen wants `{ prefs, pitch, collect }`.
MetroScreen wants `{ prefs }`. DeckScreen wants `{ prefs }`. The
coordinator gives one entry point and one return shape; screens destructure.

### 2.7 Dependency graph

```
useUiPrefsStore  ──┬─→ useAudioStream    (reads hiFiMode)
                   ├─→ usePitchPipeline  (reads filterMode, gainMode, lowCutDb, peakLock, instrumentKey, refHz)
                   └─→ useCollectBuckets (reads peakLock, instrumentKey, refHz)

useAudioStream   ───→ usePitchPipeline   (provides buffer stream)

usePitchPipeline ───→ useCollectBuckets  (provides lastFrameRef, qualityHistoryRef)
```

No cycles. Sibling hooks never reach across — everything flows downstream
through the coordinator. The only shared-mutable concession is
`qualityHistoryRef` between §2.3 and §2.4, owned by the coordinator and
threaded into both.

**Pref fields that DO NOT move** (intentional non-moves):
- `metroTimeSigKind`, `metroTimeSigPreset`, `metroCustomNumerator`,
  `metroCustomDenominator`, `metroPatternJson`, `metroSubdivisions`,
  `metroSubdivisionVoiceMidi`, `metroSubdivisionVoiceVelocity` —
  these are **scheduler state masquerading as prefs**. The scheduler
  in useMetronome owns the live values; the AsyncStorage write is a
  side-effect of the setter. Moving them out adds a hop without saving
  identity churn (none of those values ever change at audio rate; they
  change only on user touch, exactly like prefs would).
  **Compromise**: useMetronome may call `prefs.update({ metroPatternJson:
  JSON.stringify(p) })` to ride the shared debounce, but the live state
  (`pattern`, `subdivisions`, etc.) stays inside the metronome hook.
  See §8.Q3.
- `firstLaunchComplete`, `lang`, `sessionTargetCents`, `a4Hz` (legacy),
  `audioSampleRate` — these are either unused or owned elsewhere
  (a4Hz/refHz collapse — see §8.Q4).

### 4.2 Return surface

```ts
export interface UiPrefsStore {
  prefs: UiPrefs;                   // Read-only snapshot.
  prefsLoaded: boolean;             // False until first loadPrefs() resolves.
  update: (patch: Partial<UiPrefs>) => void;   // Write-through + debounced persist.
  flush: () => Promise<void>;       // Force any pending write to land. Called by AppState 'background'.
  // Convenience setters for the most common single-field updates — all just
  // call update() under the hood but keep call sites compact.
  setTheme: (t: ThemeName) => void;
  setNightDarken: (v: number) => void;
  // ... one per field for ergonomics. All thin wrappers around update().
}
```

**Why `update(patch)` plus per-field setters**: the patch form is the
canonical write path (single debounced AsyncStorage write per batch). The
per-field setters exist so existing call sites
(`engine.setTunerStyle('arc')`) translate to `prefs.setTunerStyle('arc')`
with a one-token rename. Once §5's migration completes, per-field setters
could be deprecated in favor of `update({ tunerStyle: 'arc' })`, but
that's a v1.4 concern.

### 4.3 Persistence: single debounce, one writer

Today, six different code paths call `savePrefs` independently:
- `useAudioEngine.savePrefsNow` (line 820)
- `useAudioEngine.setLowCutDb`, `setPeakLock`, `setTheme`, `setNightDarken`,
  `setNightWarmth`, `setTunerStyle`, `setMetroStyle`, `setDeckStyle`,
  `setMetroClickOffsetMs`, `setMetroOutputRoute`, `setHiFiMode` (~12 inline IIFEs)
- `useMetronome.setTimeSig`, `setBeatInstrument`, `setSubdivisions`,
  `setSubdivisionVoice`, `setClickVolume` (5 IIFEs)
- `App.tsx.persistDronePref`, `handleSetShowDebugOverlay`, AppState 'background'
- `useAudioEngine`'s own AppState 'background' listener
- The hydration race in `setHiFiMode` (does a read-then-write)

This is fragile. Two near-simultaneous setter calls can produce a
read-modify-write loop where one save loses fields the other just wrote
(today the engine's `savePrefsNow` mitigates this by re-reading current
prefs first — but the metronome setters don't).

**Replace with**: `useUiPrefsStore.update(patch)`:
1. Updates in-memory state immediately (synchronous setState).
2. Merges `patch` into `pendingSaveRef.current`.
3. Resets a 250 ms debounce timer. On fire, writes the merged
   `{ ...currentSnapshot, ...pending }` to AsyncStorage. Clears
   `pendingSaveRef`.
4. AppState 'background' fires `flush()` synchronously (clears the timer,
   does the write immediately).

This kills the read-modify-write race entirely (only one writer) and
reduces AsyncStorage write rate under rapid setter calls (e.g. the
`metroClickOffsetMs` stepper).

## 5. Migration plan

### 5.1 Order of operations

**Phase 1 — Extract `useUiPrefsStore` first.** This is the biggest
identity-stability win for the least architectural risk. The other hooks
keep working unchanged during the transition because we keep
`useAudioEngine`'s existing fields as a thin pass-through that reads from
`prefs`. Once all consumers migrate to read from `prefs` directly,
`useAudioEngine`'s pref fields are dropped.

**Phase 2 — Split `useAudioEngine` into `useAudioStream` +
`usePitchPipeline` + `useCollectBuckets`.** Done in one commit (they're
too coupled to split incrementally — see §7) but the OLD `useAudioEngine`
remains as a re-aggregating wrapper for one more release so screens
keep compiling.

**Phase 3 — Delete the `useAudioEngine` wrapper, migrate all screens.**

This is the safe order because Phase 1 is screen-visible (the screen
imports change) but logic-invisible (same data, same behavior), and Phase
2 is logic-internal (the import surface stays the same).

### 5.2 Backward compatibility per phase

- **Phase 1**: `useAudioEngine()` keeps returning every field it does today.
  Internally, the pref fields are now `useUiPrefsStore`-backed pass-throughs.
  All current call sites compile unchanged. App.tsx ALSO calls
  `useUiPrefsStore()` directly for the new screens that need it (the test
  case is moving drone prefs out — App.tsx switches to `prefs.droneVoice`
  while `engine.droneVoice` doesn't exist anyway, so this is a pure-add).
- **Phase 2**: `useAudioEngine()` becomes a 30-line composition wrapper:
  ```ts
  export function useAudioEngine(): AudioEngineState {
    const prefs   = useUiPrefsStore();
    const stream  = useAudioStream(prefs);
    const pitch   = usePitchPipeline(stream, prefs);
    const collect = useCollectBuckets(pitch, prefs);
    return useMemo(() => ({ ...prefs.prefs, ...stream, ...pitch, ...collect, /* setters */ }), [...]);
  }
  ```
  This memo will STILL produce new refs at audio rate (because `pitch`
  contains `meterFill`), but screens that have already migrated to read
  `prefs`/`stream`/`collect` directly won't go through this wrapper.
  Only the not-yet-migrated screens still pay the cost.
- **Phase 3**: wrapper deleted. All screens import the split hooks.

### 5.3 Risk assessment

| Phase | Risk | Mitigation |
|---|---|---|
| 1 | Pref read/write semantics change subtly (race on the existing inline IIFEs vs. the new debounced writer) | Audit all 18 setter paths against the new `update()` contract before deleting the old IIFE; add a smoke test that calls setTunerStyle 10× in a tight loop and verifies the persisted value matches the last call. |
| 2 | The `qualityHistoryRef` shared ref between `usePitchPipeline` and `useCollectBuckets` is the one shared-mutable in the design. Risk: lifecycle mismatch (one hook unmounts before the other, or both reset on remount inconsistently). | Own the ref in the `useAudio` coordinator (§2.6). Thread into both hooks as a prop. Coordinator unmount tears down everything atomically. |
| 2 | Splitting the RAF tick (pump now lives in §2.4, vote loop in §2.3) introduces a potential frame skew | Solution: keep ONE RAF in `usePitchPipeline`. The pipeline calls `collect.pump(latestFrameMs)` synchronously at the end of each tick. No separate RAF in §2.4. |
| 2 | Drone-chase wiring split (§2.5 option a) duplicates state across hooks | Fold drone-chase into `usePitchPipeline` (§2.5 option b). One hook owns the refs and the suspicion counter. |
| 3 | Screen migration is mechanical but touches all four screens at once | Land Phase 3 as a single commit per screen (4 small PRs), each verifying the screen renders identically before and after. Keep the `engine` prop drilling alongside the new `prefs` prop until all four screens land. |

## 6.5 Native MIDI-bus layer (v1.3.0 MUST)

This section is architecturally required, per coordinator directive. It
sits between the consumer hooks (§2) and `@local/raw-audio-output`. The
existing v1.2 state-machines SPEAK MIDI (they call `synth.noteOn`,
`synth.programChange`, `synth.pitchBend` against a singleton) but they do
NOT feel like MIDI — there is no single owner of the synth lifecycle, no
channel-role discipline, and no diagnostic surface. v1.3.0 fixes that
with a `useMidiBus` layer.

### 6.5.1 Why now

Today, three hooks reach directly into the singleton (`@local/raw-audio-
output`):

- `useDrone` (channel 0): full lifecycle (prepareAsync, start), master gain,
  noteOn/Off, pitchBend, programChange. Considers itself the de-facto owner.
- `useMetronome` (channel 9): also calls prepareAsync (idempotent — both hooks
  race), noteOn/Off on drums.
- Future v1.3+: pitch pipes from TUNER tab, aux voices, etc.

Symptoms:
1. **No single owner of master gain.** The drone hook treats `setMasterGain`
   as its private knob (used for background-mute and duck envelope). When
   useDeck mutes during recording (`App.tsx:458-460`), it goes through
   `drone.setMuted(true)` — but that ONLY mutes the drone path; any metronome
   noteOn during recording still bleeds through.
2. **No per-channel busy state.** A future hook that decides to use channel
   0 for, say, a transient pitch reference would silently fight useDrone.
3. **Two `prepareAsync` calls on cold start.** Both hooks call it
   independently. The native side is idempotent per Gandalf's contract
   (line 213 in useDrone), but they're racing with no coordination — and
   useDrone reapplies state once ready, while useMetronome polls each beat.
   The state machines should share one ready signal.
4. **Zero diagnostic visibility.** A failed noteOn is silently swallowed
   inside `try { synth.noteOn(...) } catch { /* ignore */ }` in 14 places
   across useDrone and useMetronome. The forensic logger (`src/log.ts`)
   doesn't see any MIDI activity. Debugging "why is the drone silent" is
   guess-and-check.

### 6.5.2 Bus API surface

New file `/home/repro/code/bellcurve/src/useMidiBus.ts`:

```ts
export type ChannelRole = 'drone' | 'drums' | 'pipes' | 'aux1' | 'aux2' | 'aux3' | 'aux4';

/** GM-respecting channel assignments. Drums MUST stay 9 (GM channel 10). */
export const CHANNEL_FOR_ROLE: Record<ChannelRole, number> = {
  drone: 0,
  drums: 9,    // GM percussion — non-negotiable
  pipes: 1,
  aux1: 10,
  aux2: 11,
  aux3: 12,
  aux4: 13,
};

export interface MidiChannelHandle {
  readonly role: ChannelRole;
  readonly channel: number;        // resolved GM channel index
  /** Set patch (program). Returns false if synth not ready or channel released. */
  setProgram(program: number): boolean;
  /** noteOn at velocity 0..1. clickVolume/masterMute multipliers applied by bus. */
  noteOn(midi: number, velocity: number): boolean;
  noteOff(midi: number): boolean;
  pitchBend(semitones: number): boolean;   // clamped to ±12 by bus
  allNotesOff(): void;
  /** Release the channel. Future reserve() can re-acquire. */
  release(): void;
}

export interface MidiBus {
  /** True once synth.prepareAsync resolved successfully. Reactive — re-renders consumers. */
  ready: boolean;
  /** False until the WAV-fallback bound (MAX_FALLBACK_BARS-style) is exceeded for the drum-WAV degraded mode. */
  warmingUp: boolean;
  /**
   * Reserve a channel role. Returns null if already in use AND `opts.steal !== true`.
   * Default behavior: first caller wins; subsequent reserve() returns null and logs a warning.
   */
  reserve(role: ChannelRole, opts?: { steal?: boolean }): MidiChannelHandle | null;
  /** Diagnostic: returns the set of currently-reserved roles. */
  reservations(): readonly ChannelRole[];
  /** Global mute — useDeck calls this on record-start. Affects every channel. */
  setMasterMute(on: boolean): void;
  /** Master volume 0..1. Drone uses this for its volume slider via the channel's volume scalar — see §6.5.4 on volume math. */
  setMasterGain(gain: number): void;
  /** Subscribe to MIDI events for diagnostics + forensic logger. */
  addEventListener(fn: (e: MidiBusEvent) => void): () => void;
}

export type MidiBusEvent =
  | { kind: 'ready' }
  | { kind: 'noteOn'; role: ChannelRole; channel: number; midi: number; velocity: number; tsMs: number }
  | { kind: 'noteOff'; role: ChannelRole; channel: number; midi: number; tsMs: number }
  | { kind: 'programChange'; role: ChannelRole; channel: number; program: number; tsMs: number }
  | { kind: 'pitchBend'; role: ChannelRole; channel: number; semitones: number; tsMs: number }
  | { kind: 'underrun'; reason: string; tsMs: number }
  | { kind: 'reserved'; role: ChannelRole }
  | { kind: 'released'; role: ChannelRole };

export function useMidiBus(): MidiBus;
```

### 6.5.3 Synth singleton ownership moves to the bus

- `useMidiBus` is the **only** module that imports `@local/raw-audio-output`.
- The bus owns the lifecycle: `prepareAsync` on mount, `start` on ready,
  `addReadyListener`, AppState 'background' → `setMasterGain(0)`, foreground
  → restore, unmount → `allNotesOff` across reserved channels.
- The bus tracks per-channel reservation state and per-channel pending notes
  (so `allNotesOff` on release is precise — no global `allNotesOff` across
  channels owned by other hooks).
- `useDrone`, `useMetronome`, and any future hook DO NOT import
  `@local/raw-audio-output` directly. Lint rule: ban that import outside
  `useMidiBus.ts` (eslint `no-restricted-imports`).

### 6.5.4 Master-mute / record coordination

Today: `useDeck` toggles `drone.setMuted(true)` during recording, which
calls `synth.setMasterGain(0)` through `applyGain()` (useDrone.ts:155-161).
The metronome path is uncoordinated — if the user records while the
metronome is running, drum noteOns still fire.

Post-bus:

- `useDeck` calls `bus.setMasterMute(true)` on record-start, false on
  record-stop. Bus multiplies the master mute into every channel's effective
  velocity AND clamps `setMasterGain` to 0 while muted.
- `useDrone.setMuted` becomes a thin wrapper: `bus.setMasterMute(on)`. It
  no longer maintains its own muted ref — the bus is authoritative.
  Alternative: drop `useDrone.setMuted` entirely; let `useDeck` call the
  bus directly. Captured as §8.Q12 below.
- The drone's duck envelope (§250-277 of useDrone) moves into the bus as
  `bus.channel('drone').duck(holdMs)` — OR stays in useDrone and just calls
  through `bus` for the gain writes. Gandalf recommends the latter — the
  duck math (DUCK_DEPTH, FADE_IN_MS, FADE_OUT_MS) is drone-specific (an
  arbitrary aux channel wouldn't want the same envelope).

### 6.5.5 Volume math

The bus exposes `setMasterGain(gain)`. Per-channel scalars are applied
multiplicatively at noteOn time:

```
effectiveVelocity = noteOnVelocity * channelVolume[role] * (masterMuted ? 0 : 1)
```

useDrone owns its "channel volume" via `bus.channel('drone').setVolume(v)`
(replaces today's `targetVolumeRef` flowing into setMasterGain). This is
the only way to safely have a drone playing softly while the metronome
plays at full velocity — today both share `setMasterGain` and fight.

**Add `setVolume(v: number)` to `MidiChannelHandle`** (omitted from the
sketch above for brevity).

### 6.5.6 WAV-fallback ownership

Today useMetronome owns the WAV-fallback warm-up (lines 294-300 + 402-407
+ 487-495). The bus is the natural owner — it knows `ready` synchronously.
Two options:

**(a) Bus owns WAV fallback for drums only.** Bus exposes
`channel('drums').noteOnWithFallback(midi, vel)` that internally chooses
TSF if ready, else fires the accent/normal WAV. useMetronome stops owning
the AudioPlayer instances; bus owns them.

**(b) Bus only flips `ready` / `warmingUp` flags; useMetronome owns WAV.**
Less invasive but keeps useMetronome importing expo-audio's
`createAudioPlayer` and `buildClickWavBase64`.

Gandalf recommends **(a)** — the WAV fallback is a property of the SYNTH
layer, not the metronome. Captured as §8.Q13.

### 6.5.7 Event subscription + forensic log forwarding

`useMidiBus` instances a single listener that forwards every event to
`src/log.ts` via `log.i('MidiBus', '<kind> ch<channel> midi=<midi> v=<vel>')`.
Configurable at `useUiPrefsStore.showDebugOverlay` time — when overlay is
ON, log every event; when OFF, log only `ready`, `underrun`, `reserved`,
`released` (the rare ones).

Consumer hooks can also `bus.addEventListener` for their own UI — e.g.,
the debug overlay tab could show a live MIDI event tape.

### 6.5.10 `usePitchPipes` (channel `pipes`)

Per coordinator directive: pitch pipes ALSO move onto the MIDI bus. They
become the third consumer hook alongside `useDrone` (channel 0,
`'drone'`) and `useMetronome` (channel 9, `'drums'`). Channel role
`'pipes'` maps to GM channel 1.

**Today's pipes path** (`src/components/PitchPipes.tsx` + `src/pitchTones.ts`):
- The modal renders a chromatic-octave pad grid.
- Tap a pad: `startTone(midi)` builds a looped WAV via
  `buildWavBase64(midi, refHz)`, writes to cache, creates an
  `AudioPlayer`, sets `loop=true`, plays.
- Tap the same pad again: `stopTone()` calls `player.remove()`.
- Tap another pad while one is playing: stops the first, starts the new
  one. Single-voice semantics.
- Closing the modal or changing `refHz` stops the tone.
- `pitchTones.ts` exists solely to synthesize a WAV at the user's A4
  reference. Once the bus pipes the same note through TSF, that file
  becomes unused (delete in v1.4 or keep around for offline diagnostic
  generation — see §8.Q15).

**New `usePitchPipes` hook** (new file
`/home/repro/code/bellcurve/src/usePitchPipes.ts`):

```ts
export interface PitchPipesState {
  /** MIDI of the currently sounding pipe, or null if silent. */
  playingMidi: number | null;
  /** Tap-to-toggle one pipe. Same midi as currently playing → silence. */
  toggle: (midi: number) => void;
  /** Force stop (e.g. modal dismissed). */
  stop: () => void;
  /** Voice id from the same DroneVoice catalog (see §6.5.10 below). */
  voice: string;
  setVoice: (id: string) => void;
}

export interface UsePitchPipesArgs {
  bus: MidiBus;
  /** A4 reference — drives the baseline pitch bend so pipes track refHz. */
  a4Hz: number;
}

export function usePitchPipes(args: UsePitchPipesArgs): PitchPipesState;
```

**Behavior**:

- **Channel reserve**: lazy. The hook reserves `'pipes'` on first
  `toggle()`, NOT on mount. (No reservation while the modal is closed.)
  Releases on `stop()`. This keeps the bus's `reservations()` set lean
  and lets a future hook briefly borrow channel 1 if pipes are idle.
- **Infinite sustain**: `toggle(midi)` issues
  ```
  handle.setProgram(voice.program);
  handle.noteOn(midi, 1.0);
  handle.pitchBend(a4BendSemitones(a4Hz));
  ```
  and **does NOT schedule a noteOff**. The note holds indefinitely. The
  next `toggle(midi)` on the same midi → `handle.noteOff(midi)`. The next
  `toggle(otherMidi)` → `handle.noteOff(previousMidi); handle.noteOn(otherMidi, 1.0)`.
- **A4 baseline bend**: applied after every noteOn AND whenever `a4Hz`
  changes while a pipe is sounding. Same formula as
  `useDrone.a4BendSemitones`:
  ```ts
  function a4BendSemitones(a4Hz: number): number {
    if (!Number.isFinite(a4Hz) || a4Hz <= 0) return 0;
    return 1200 * Math.log2(a4Hz / 440) / 100;
  }
  ```
  Clamped to ±12 by the bus's `pitchBend()`. (A4 between 430-450 Hz
  resolves to roughly ±0.4 semitones — well inside the bend range.)
- **Default voice**: `'choir-aahs'` (GM program 52). Warm, in-tune,
  clearly pitched, sustained. Listed in `droneVoices.ts` already (or
  added there — see §6.5.10 below for the shared catalog).
- **Modal lifecycle coordination**: `PitchPipes.tsx` already calls
  `stopTone()` on close and on refHz change. Post-refactor, those become
  `pipes.stop()` and `pipes.toggle()` honors a fresh `a4Hz` baseline bend
  via the dep array. The hook owns the cleanup; the modal becomes a
  dumber view.
- **Mute-on-record**: nothing pipes-specific. `bus.setMasterMute(true)`
  silences channel 1 along with channels 0 and 9. The pipe note keeps
  sounding (held), it just goes inaudible.
- **AppState background**: bus's master gain drops to 0 (already covered).
  When the app foregrounds, the pipe note resumes IF the user hadn't
  toggled it off. Symmetric with drone behavior.

## 7. What NOT to split

Things that look tempting but would create more problems than they solve.

### 7.1 YIN + filter state + ring buffer + display RAF

These ARE coupled. The buffer cadence (`bufferDurationMsRef`) is set by
the audio stream, consumed by the filter (`filterMode` window in ms →
samples), and consumed by the display tick (RESPONSE-window-in-ms →
sample count). All three live in one mental coordinate system. The
desktop's filter hysteresis (`filterModes.ts`) was hand-tuned against
the buffer-rate cadence — touching that coupling risks regressing the
filter behavior in subtle ways that don't surface until a user with an
older device reports "the tuner stopped reading my low Bb."

Keep YIN, filter, ring buffer, AND the display RAF inside
`usePitchPipeline`. The boundary to draw is between "audio frame
in / display state out" and the consumers (COLLECT buckets, drone, screens).

### 7.2 incumbentMidi from freqHz

Tempting: "incumbentMidi is a clean signal; useDrone wants it; carve it
off." But incumbentMidi is computed in the SAME RAF tick that produces
freqHz from the median-of-incumbent-freqs (§1895-1913 of useAudioEngine).
The two values must be consistent within a frame (drone hears the rounded
note, screen reads the median freq that voted for that rounded note).
Splitting risks a 16ms skew where the drone is sounding the new note but
freqHz is still showing the previous one. Keep them in `usePitchPipeline`.

### 7.3 Quality gate from pitch detection

The gate's transient filter (#1) reads df/dt — i.e. it needs the previous
frame's exact MIDI. If the gate lives in a separate hook and the pipeline
just passes "here's this frame, you decide," the gate works. But the gate
also needs to update its history struct EVERY frame, even on rejection
(so the rms ring stays warm). Splitting via callback means the pipeline
calls `gate.evaluate(frame)` on every audio buffer — a function call
across a hook boundary at 40 Hz. That's fine in practice but the
qualityHistoryRef ownership question (§2.4 + §5.3) is the real cost.

**Decision**: gate evaluation stays inside `usePitchPipeline.onBuffer`
(via the shared `qualityHistoryRef` owned by the coordinator). The
BUCKET accumulation logic moves to `useCollectBuckets`. Pipeline writes
`{ decision, frameSummary }` into a `lastFrameRef`; collect reads it
inside its pump.

### 7.4 useMetronome.pattern / subdivisions / pattern-related state

The scheduler reads `patternRef`, `subdivisionsRef`, `subdivisionVoiceRef`
on every beat dispatch (`useMetronome.ts:431-477`). These need to be
co-located with the scheduler so the ref-write happens in the same
component tree. Moving them out to prefs adds a sync hop without solving
anything (they're already user-rate). Leave them in useMetronome.

### 7.5 useDeck's player + recorder

Two `useAudio*` hook instances live inside useDeck (`useAudioRecorder` +
`useAudioRecorderState`) and a manual `AudioPlayer` ref. The temptation
is "extract a `usePlaybackController`." Don't. The lifecycle
(recordingClosingRef + AppState background race-guard) is tightly
coordinated between record stop and player initialization. Splitting
re-introduces the v1.0.1 BUG-5 single-shot guard issue. Leave intact.
